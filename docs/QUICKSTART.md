# Rusternetes Quick Start

Get a Rust-based Kubernetes cluster running locally.

## Prerequisites

- **Rust toolchain** -- install via [rustup](https://rustup.rs/)
- **Docker Desktop** (macOS) or **Docker Engine** (Linux)
- **Docker Compose** (included with Docker Desktop; install separately on Linux if needed)
- **kubectl** (optional) -- standard `kubectl` works against the cluster

## Start the Cluster

```bash
# Clone and build
git clone https://github.com/calfonso/rusternetes.git
cd rusternetes
cargo build --release

# Set the volumes path (required for kubelet bind mounts)
export KUBELET_VOLUMES_PATH=$(pwd)/.rusternetes/volumes

# Build container images (~10-15 minutes the first time)
docker compose build

# Start all services
docker compose up -d

# Bootstrap CoreDNS, default services, and ServiceAccount tokens
bash scripts/bootstrap-cluster.sh
```

## Use the Cluster

```bash
export KUBECONFIG=~/.kube/rusternetes-config

kubectl get nodes
kubectl get pods -A
kubectl get services -A
```

The KUBECONFIG file is generated by the bootstrap script. Standard `kubectl`
works because the API server speaks the same REST API as upstream Kubernetes.

## What You Get

| Component            | Details                                      |
|----------------------|----------------------------------------------|
| etcd                 | Port 2379, cluster state storage             |
| API Server           | Port 6443, HTTPS with auto-generated TLS     |
| Scheduler            | Affinity, taints/tolerations, preemption     |
| Controller Manager   | 31 controllers (Deployment, StatefulSet, Job, DaemonSet, HPA, etc.) |
| Kubelet (node-1)     | Container runtime via Docker/bollard         |
| Kubelet (node-2)     | Second worker node                           |
| Kube-Proxy           | iptables-based ClusterIP and NodePort routing (host network) |
| CoreDNS              | Kubernetes-compatible service discovery      |

TLS certificates are auto-generated in `.rusternetes/certs/` by
`scripts/generate-certs.sh` (called during the Docker build).

## Common Operations

### View Logs

```bash
docker compose logs -f api-server
docker compose logs -f node-1
docker compose logs -f scheduler
docker compose logs -f controller-manager
```

### Rebuild After Code Changes

```bash
# Rebuild a single service
docker compose build api-server
docker compose up -d api-server

# Rebuild everything
docker compose build
docker compose up -d
```

### Stop the Cluster

```bash
docker compose down
```

### Clean Restart (wipes etcd state)

```bash
docker compose down -v
docker compose up -d
bash scripts/bootstrap-cluster.sh
```

### Run Tests

```bash
# Unit tests (all crates)
cargo test

# Single crate
cargo test -p rusternetes_api_server

# Lint and format
cargo fmt --all
cargo clippy --all-targets --all-features -- -D warnings

# All checks before committing
make pre-commit
```

### Run Conformance Tests

```bash
bash scripts/run-conformance.sh
bash scripts/conformance-progress.sh    # monitor pass/fail progress
```

E2e output lives at `/tmp/sonobuoy/results/e2e.log` inside the e2e container.

## Troubleshooting

### "Port already in use"

```bash
lsof -i :6443
lsof -i :2379
# Kill the conflicting process, or stop any other local K8s cluster
```

### "Cannot connect to API server"

```bash
# Check that services are running
docker compose ps

# Restart everything
docker compose down
docker compose up -d
bash scripts/bootstrap-cluster.sh
```

### "Build is taking too long"

The first `docker compose build` compiles all Rust crates inside Docker and
takes 10-15 minutes. Subsequent builds use the Docker layer cache and are
much faster. For local iteration without Docker, use `cargo build` directly.

## What's Different from Real Kubernetes

Rusternetes is a ground-up reimplementation of Kubernetes in Rust. It is not
a fork -- every component is written from scratch.

- Written entirely in Rust (upstream Kubernetes is Go)
- Uses Docker as the container runtime via the bollard library
- Standard `kubectl` works against it (same REST API surface)
- Currently passes ~60% of the official Kubernetes conformance tests (328 fixes applied, actively improving)
- Educational and development use -- not production-hardened

**Use it to** learn how Kubernetes works internally, experiment with Rust for
systems programming, or study distributed system design.

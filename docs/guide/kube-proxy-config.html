Kube-Proxy Configuration

Kube-proxy implements service networking through iptables rules. It runs in host network mode and requires elevated capabilities to manage the host's NAT tables.

CLI Flags

Flag Type Default Description
--node-name String required Name of this node
--etcd-servers String http://localhost:2379 Comma-separated list of etcd endpoints
--storage-backend String etcd Storage backend: etcd or sqlite
--data-dir String ./data/rusternetes.db SQLite database path (only with --storage-backend=sqlite)
--log-level String info Log level: trace, debug, info, warn, error
--sync-interval u64 1 Sync interval in seconds for the primary watch-triggered loop

Required Capabilities

Kube-proxy must run with elevated privileges to manage iptables rules on the host:

Requirement Purpose
CAP_NET_ADMIN Required for iptables rule manipulation
CAP_NET_RAW Required for raw socket operations
CAP_SYS_ADMIN Required for network namespace operations
Host network mode Must share the host's network namespace to manage host iptables 
        

          
          
          
          docker-compose.yml
        

        kube-proxy:
  network_mode: host
  cap_add:
    - NET_ADMIN
    - NET_RAW
    - SYS_ADMIN

iptables Mode

Kube-proxy uses iptables mode exclusively (the only mode implemented). It manages four chains in the NAT table:

Chain Purpose
KUBE-SERVICES Entry point. Matches traffic destined for ClusterIPs and dispatches to per-service chains.
KUBE-SVC-* Per-service chain. Load-balances across backend endpoints using probability-based rules.
KUBE-SEP-* Per-endpoint chain. DNATs traffic to the actual pod IP and port.
KUBE-FORWARD Filter table chain. Accepts RELATED,ESTABLISHED return traffic and forwarded packets. 
        

          
          
          
          iptables flow
        

        # NAT table (PREROUTING + OUTPUT)
Packet --> KUBE-SERVICES --> KUBE-SVC-XXXX --> KUBE-SEP-YYYY --> DNAT to pod

# Filter table (FORWARD)
Packet --> KUBE-FORWARD --> ACCEPT (RELATED,ESTABLISHED)
                         --> ACCEPT (forwarded traffic)
RELATED,ESTABLISHED

The KUBE-FORWARD chain must accept RELATED,ESTABLISHED return traffic in the filter table. Without this, response packets from backend pods are dropped, causing "Connection refused" errors. The OUTPUT chain also needs a jump to KUBE-FORWARD for locally-originated traffic.

Service Types

Type iptables Rules
ClusterIP DNAT rules for the virtual ClusterIP to backend pod IPs
NodePort ClusterIP rules plus additional rules matching traffic on the NodePort across all interfaces
LoadBalancer ClusterIP + NodePort rules plus rules for the external LoadBalancer IP
ExternalName No iptables rules (DNS-only, skipped by kube-proxy)

Session Affinity

When a service has spec.sessionAffinity: ClientIP, kube-proxy adds iptables --recent rules to route subsequent connections from the same client IP to the same backend. The timeout defaults to 10800 seconds (3 hours), configurable via spec.sessionAffinityConfig.clientIP.timeoutSeconds.

State Hashing

To avoid unnecessary iptables rebuilds, kube-proxy computes a hash of the current service/endpoint state and compares it to the previous sync. If the hash is unchanged, the sync is skipped entirely.

ORDER-INDEPENDENT HASHING

The state hash uses XOR of per-item hashes to be order-independent. etcd returns items in arbitrary order, so a naive ordered hash would trigger unnecessary iptables flush+rebuild on every sync cycle even when nothing changed.

Atomic Updates

Kube-proxy attempts to apply NAT rules atomically via iptables-restore --noflush. This avoids the "Connection refused" window that occurs when flushing all rules and rebuilding them one by one. If the atomic restore fails, it falls back to flush+rebuild as a safety measure.

Resync Safety Net

In addition to the watch-triggered sync loop, kube-proxy runs a periodic resync every 15 seconds as a safety net. This catches any changes that might have been missed due to watch stream interruptions or network issues. 

        

          
          
          
          sync triggers
        

        # Primary: watch-triggered (immediate)
Service watch event    -->  sync
Endpoints watch event  -->  sync

# Secondary: periodic safety net
Every 15 seconds       -->  sync (skipped if hash unchanged)

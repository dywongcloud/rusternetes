Web Console

A built-in web dashboard served directly from the API server — no proxy, no separate deployment, no external dependencies. Browse workloads, nodes, networking, storage, RBAC, and events in real time. Supports multi-cluster management for fleets of rusternetes instances.

EMBEDDED BY DESIGN

The console is served by the same Axum HTTP server that handles the Kubernetes API. When you pass --console-dir, the API server mounts a ServeDir handler at /console/ with an index.html fallback for client-side routing. No nginx, no OAuth proxy, no CORS — just one process, one port.

Quick Start

Build the console SPA and start the API server with it enabled:

        

          
          
          
          Terminal
        

        # Build the console
cd console
npm install
npm run build

# Start rusternetes with the console enabled
cd ..
./target/release/rusternetes --console-dir ./console/dist

# Open in browser
open https://localhost:6443/console/

Development Mode

For hot-reloading during console development, run the Vite dev server alongside the API server:

        

          
          
          
          Terminal
        

        # Terminal 1: start the API server (no --console-dir needed)
./target/release/rusternetes

# Terminal 2: start the console dev server
cd console
npm run dev
# Vite on http://localhost:3000/console/
# Proxies /api and /apis to localhost:6443

Architecture

        

          
          
          
          Architecture
        

        Browser ── https://localhost:6443/console/
               │
               ├── /console/              Static SPA files (Axum ServeDir)
               ├── /api/v1/pods           K8s REST API (same server)
               ├── /api/v1/pods?watch=1   Watch stream (chunked HTTP)
               └── /apis/apps/v1/...      API group resources

Because the SPA and API share the same origin, there is no CORS configuration needed. The console calls the standard Kubernetes REST API endpoints directly using fetch().

Views

The console provides 8 views accessible from the sidebar navigation:

Overview

Cluster dashboard with stat cards showing pod counts (running/total), node status (ready/total), namespace count, and discovered resource types. Below the stats is a deployment summary showing ready replica counts and availability status.

Workloads

Tabbed table view for the core workload types:

  • Pods — name, namespace, status, ready containers, restart count, node, age
  • Deployments — name, namespace, ready/desired replicas, up-to-date, available, age
  • StatefulSets, DaemonSets, Jobs — name, namespace, age

Networking

Tabbed view for:

  • Services — name, namespace, type, ClusterIP, ports
  • Endpoints, Ingresses, Network Policies

Nodes

Table showing node name, Ready status, roles (from node-role.kubernetes.io/ labels), kubelet version, OS/architecture, CPU and memory capacity, and age.

Storage

Tabbed view for PersistentVolumeClaims (status, capacity, access modes, storage class), PersistentVolumes (status, capacity, reclaim policy), and StorageClasses.

Config

Tabbed view for ConfigMaps (data key count), Secrets (type, key count), and ServiceAccounts.

RBAC

Tabbed view for ClusterRoles (rule count), ClusterRoleBindings (role ref, subjects), Roles, and RoleBindings. Cluster-scoped resources ignore the namespace filter; namespaced resources respect it.

Events

Live event stream with 10-second auto-refresh. Events are sorted newest-first, color-coded by type (Normal=blue, Warning=amber, Error=red), and display the involved object, reason, message, timestamp, and occurrence count.

K8s API Client

The console communicates with the API server using the standard Kubernetes REST protocol. No custom API endpoints are required.

Resource Discovery

On load, the console fetches /api/v1 and /apis to discover all available resource types, including CRDs. Results are cached for 5 minutes. The resource registry drives the namespace selector, view routing, and table column configuration.

Watch Streams

The WatchManager class implements the standard K8s watch protocol:

  • Chunked HTTP responses with newline-delimited JSON
  • resourceVersion tracking and BOOKMARK support
  • Automatic reconnection with exponential backoff (1s → 30s max)
  • 410 Gone recovery (resets resourceVersion and re-lists)

Authentication

The console reads a JWT token from sessionStorage and passes it as an Authorization: Bearer header. In --skip-auth mode (development), no token is required.

Multi-Cluster

The console includes a fleet mode for managing multiple rusternetes clusters from a single browser tab.

How It Works

  1. Click the Fleet button in the header to enable multi-cluster mode
  2. Click + to register a remote cluster (name + API server URL)
  3. Click a cluster name to switch the active context
  4. All views and API calls automatically route to the selected cluster
        

          
          
          
          Multi-Cluster Architecture
        

        Browser → rusternetes-hub (Axum)
              ├── /api/v1/...               Local cluster API
              ├── /console/                 SPA static files
              └── /clusters/{id}/api/...    Proxy to remote cluster

For the local cluster, API calls are same-origin (no prefix). For remote clusters, the console prefixes paths with /clusters/{cluster-id}, and the hub proxies requests to the remote cluster's API endpoint. Cluster registrations persist in the browser's localStorage.

CLI Reference

Flag Default Description
--console-dir disabled Path to the console SPA build directory. Enables the web console at /console/.

When --console-dir is not set, the API server does not serve any console routes.

Building from Source

        

          
          
          
          Terminal
        

        cd console

# Install dependencies
npm install

# Type-check
npm run type-check

# Development server (hot reload)
npm run dev

# Production build (~313KB JS + ~13KB CSS, gzipped ~96KB)
npm run build

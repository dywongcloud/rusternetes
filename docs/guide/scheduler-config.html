Scheduler Configuration

The scheduler assigns unscheduled pods to nodes using a filter/score plugin framework. It supports leader election for high-availability deployments.

CLI Flags

Flag Type Default Description
--etcd-servers String http://localhost:2379 Comma-separated list of etcd endpoints
--storage-backend String etcd Storage backend: etcd or sqlite
--data-dir String ./data/rusternetes.db SQLite database path (only with --storage-backend=sqlite)
--log-level String info Log level: trace, debug, info, warn, error
--interval u64 2 Scheduling loop interval in seconds
--metrics-port u16 8081 Prometheus metrics server port
--enable-leader-election bool false Enable leader election for HA
--leader-election-identity String auto-generated UUID Unique identity for this scheduler instance
--leader-election-lock-key String /rusternetes/scheduler/leader etcd key used for the leader election lock
--leader-election-lease-duration u64 15 Leader election lease duration in seconds. Renew interval is lease/3.

Usage Examples

        

          
          
          
          terminal
        

        # Single instance (no leader election)
rusternetes-scheduler \
  --etcd-servers http://etcd:2379 \
  --interval 2

# HA with leader election
rusternetes-scheduler \
  --etcd-servers http://etcd:2379 \
  --enable-leader-election \
  --leader-election-identity scheduler-1 \
  --leader-election-lease-duration 15

Built-in Plugins

The scheduler uses a two-phase approach: filter plugins eliminate ineligible nodes, then score plugins rank the remaining candidates.

Filter Plugins

Filter plugins return pass/fail for each node. A node must pass all filters to be considered schedulable.

Plugin Purpose
NodeUnschedulable Filters out nodes marked as unschedulable (spec.unschedulable: true)
TaintToleration Filters nodes with taints that the pod does not tolerate
NodeSelector Filters nodes that do not match the pod's nodeSelector labels
NodeAffinity Filters nodes based on requiredDuringSchedulingIgnoredDuringExecution node affinity rules
PodAffinity Filters nodes based on pod affinity requiredDuringScheduling rules
PodAntiAffinity Filters nodes based on pod anti-affinity requiredDuringScheduling rules
TopologySpreadConstraints Filters nodes that would violate topology spread constraints with DoNotSchedule policy

Score Plugins

Score plugins assign a numeric score to each eligible node. The node with the highest total weighted score is selected.

Plugin Weight Purpose
NodeResourcesFit 25 Scores nodes based on available CPU and memory resources (least-requested)
NodeAffinity default Scores nodes matching preferredDuringScheduling node affinity rules
PodAffinity default Scores nodes matching preferredDuringScheduling pod affinity rules
PodAntiAffinity default Scores nodes matching preferredDuringScheduling pod anti-affinity rules
TopologySpreadConstraints default Scores nodes based on topology spread balance for ScheduleAnyway constraints

Scheduling Flow

        

          
          
          
          scheduling pipeline
        

        # 1. List unscheduled pods (spec.nodeName is empty, phase=Pending)
# 2. Sort by priority (highest first)
# 3. For each pod:
   Filter phase     # Run all filter plugins, eliminate ineligible nodes
   Score phase      # Run all score plugins, weight and sum scores
   Select node      # Pick the highest-scoring node
   Bind             # Set pod.spec.nodeName and update storage

# 4. If no node passes filters:
   Preemption       # Try to evict lower-priority pods to make room

Preemption

When no node passes all filter plugins for a pending pod, the scheduler attempts preemption. It evaluates whether evicting lower-priority pods from any node would make the pod schedulable:

  • Only pods with lower priority than the pending pod are candidates for eviction
  • The scheduler picks the node that requires the fewest evictions
  • Evicted pods are deleted and their owners (ReplicaSet, etc.) recreate them
  • PodDisruptionBudgets are respected during preemption
LEADER ELECTION

When leader election is enabled, the scheduler starts in follower mode and waits to acquire the lease before scheduling. If it loses the lease, it stops scheduling and waits to re-acquire. The lease is renewed every lease_duration / 3 seconds (default: 5s).

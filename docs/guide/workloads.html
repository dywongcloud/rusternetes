Workloads

Rusternetes supports the full range of Kubernetes workload resources: Pods, Deployments, ReplicaSets, StatefulSets, DaemonSets, Jobs, CronJobs, and the legacy ReplicationController. Each is managed by a dedicated controller in the controller-manager.

Pod

A Pod is the smallest deployable unit — one or more containers that share networking, storage, and a lifecycle. Pods are rarely created directly; they are usually managed by a higher-level controller such as a Deployment or Job.

Key Spec Fields

FieldTypeDescription
containersarrayRequired. List of application containers to run in the pod.
initContainersarrayContainers that run to completion before app containers start. Run sequentially — each must succeed before the next begins.
restartPolicystringAlways (default), OnFailure, or Never. Controls whether the kubelet restarts exited containers.
terminationGracePeriodSecondsintegerSeconds to wait after SIGTERM before SIGKILL. Default 30.
serviceAccountNamestringServiceAccount to use. Defaults to default in the pod's namespace. Projected token auto-injected.
hostNetworkbooleanUse the host's network namespace instead of a pod-private one.
dnsPolicystringClusterFirst (default, uses CoreDNS), Default (host DNS), ClusterFirstWithHostNet, or None (requires dnsConfig).
nodeNamestringBypass the scheduler and assign the pod to a specific node.
nodeSelectormapSimple label-match constraints for node selection.
tolerationsarrayAllow scheduling onto nodes with matching taints.
volumesarrayList of volumes that containers can mount. See Storage & Volumes.
securityContextobjectPod-level security settings (runAsUser, fsGroup, etc.). See Security.
priorityClassNamestringName of a PriorityClass for scheduling priority and preemption.

Pod Phases

PhaseMeaning
PendingAccepted but not yet scheduled or containers not yet started.
RunningAt least one container is running.
SucceededAll containers terminated successfully (exit code 0).
FailedAt least one container terminated with a non-zero exit code.
UnknownCannot determine pod state (usually a communication error with the kubelet).

Example

        

          
          
          
          pod.yaml
        

        apiVersion: v1
kind: Pod
metadata:
  name: nginx
  namespace: default
spec:
  containers:
    - name: nginx
      image: nginx:1.27
      ports:
        - containerPort: 80
      resources:
        requests:
          cpu: 100m
          memory: 128Mi
        limits:
          cpu: 500m
          memory: 256Mi
  restartPolicy: Always
  terminationGracePeriodSeconds: 30
      
        

          
          
          
          terminal
        

        # Create a pod
kubectl apply -f pod.yaml

# List pods with wide output
kubectl get pods -o wide

# Watch pod status in real time
kubectl get pods -w

# Describe a pod (events, conditions, volumes)
kubectl describe pod nginx

# Delete a pod
kubectl delete pod nginx

Deployment

A Deployment manages a set of identical pods through ReplicaSets. It provides declarative updates, rollback, scaling, and rolling update strategies. This is the most common way to run stateless applications.

Key Spec Fields

FieldTypeDescription
replicasintegerDesired number of pod replicas. Default 1.
selectorLabelSelectorRequired. Must match .spec.template.metadata.labels.
templatePodTemplateSpecPod template used to create new pods.
strategy.typestringRollingUpdate (default) or Recreate.
strategy.rollingUpdate.maxSurgeint or %Max pods above desired count during update. Default 25%.
strategy.rollingUpdate.maxUnavailableint or %Max pods that can be unavailable during update. Default 25%.
revisionHistoryLimitintegerNumber of old ReplicaSets to keep for rollback. Default 10.
minReadySecondsintegerSeconds a pod must be ready before considered available. Default 0.
pausedbooleanPause the deployment to batch multiple changes before rolling out.
progressDeadlineSecondsintegerSeconds to wait for progress before reporting failure. Default 600.

Rolling Update Behavior

With RollingUpdate strategy, the deployment controller creates a new ReplicaSet and scales it up while scaling down the old ReplicaSet. The maxSurge and maxUnavailable parameters control the pace.

maxUnavailable rounding

When maxUnavailable rounds to 0 (e.g., 25% of 1 replica), the deployment controller forces old ReplicaSets to scale down once the new ReplicaSet is fully available. This prevents a deadlock where neither RS can make progress.

With Recreate strategy, all existing pods are killed before new ones are created. This causes downtime but guarantees no two versions run simultaneously.

Example

        

          
          
          
          deployment.yaml
        

        apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: default
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  revisionHistoryLimit: 5
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 10

Rollout Commands

        

          
          
          
          terminal
        

        # Create a deployment quickly
kubectl create deployment web --image=nginx:1.27 --replicas=3

# Check rollout status
kubectl rollout status deployment/web

# View rollout history
kubectl rollout history deployment/web

# View a specific revision
kubectl rollout history deployment/web --revision=2

# Roll back to the previous revision
kubectl rollout undo deployment/web

# Roll back to a specific revision
kubectl rollout undo deployment/web --to-revision=1

# Restart all pods (triggers a new rollout)
kubectl rollout restart deployment/web

# Pause a rollout to batch changes
kubectl rollout pause deployment/web

# Resume a paused rollout
kubectl rollout resume deployment/web

# Scale the deployment
kubectl scale deployment/web --replicas=5

# Update the image (triggers rolling update)
kubectl set image deployment/web nginx=nginx:1.28

ReplicaSet

A ReplicaSet ensures a specified number of pod replicas are running at all times. ReplicaSets are rarely created directly — they are managed by Deployments. Each Deployment revision creates a new ReplicaSet.

Key Spec Fields

FieldTypeDescription
replicasintegerDesired number of replicas. Default 1.
selectorLabelSelectorRequired. Label selector for pods this RS manages.
templatePodTemplateSpecPod template for creating new replicas.
minReadySecondsintegerSeconds a pod must be ready before considered available.
Ownership

ReplicaSets created by a Deployment have an ownerReferences entry pointing back to the Deployment. The garbage collector deletes orphaned ReplicaSets when the parent Deployment is deleted with --cascade=foreground or --cascade=background.

Example

        

          
          
          
          replicaset.yaml
        

        apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: web-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
      
        

          
          
          
          terminal
        

        # List ReplicaSets
kubectl get replicasets

# Describe a ReplicaSet
kubectl describe rs web-rs

# Scale directly (prefer scaling the Deployment instead)
kubectl scale rs web-rs --replicas=5

StatefulSet

A StatefulSet manages stateful applications that need stable network identities and persistent storage. Pods get predictable names (web-0, web-1, web-2) and are created/deleted in order.

Key Spec Fields

FieldTypeDescription
serviceNamestringRequired. Name of the headless Service that provides network identity to pods.
replicasintegerDesired number of replicas. Default 1.
selectorLabelSelectorRequired. Must match .spec.template.metadata.labels.
templatePodTemplateSpecPod template. Pods are named {statefulset}-{ordinal}.
podManagementPolicystringOrderedReady (default) — create/delete sequentially. Parallel — create/delete all at once.
volumeClaimTemplatesarrayPVC templates. Each pod gets its own PVC, named {pvc}-{pod}.
updateStrategy.typestringRollingUpdate (default) or OnDelete.
updateStrategy.rollingUpdate.partitionintegerOnly pods with ordinal ≥ partition are updated. Enables canary deploys.
minReadySecondsintegerSeconds a pod must be ready before the next is updated.
ordinals.startintegerStarting ordinal (default 0). Allows non-zero-based numbering.

Stable Network Identity

Each pod in a StatefulSet gets a DNS name: {pod-name}.{service-name}.{namespace}.svc.cluster.local. For example, web-0.web-headless.default.svc.cluster.local.

Example

        

          
          
          
          statefulset.yaml
        

        apiVersion: v1
kind: Service
metadata:
  name: web-headless
spec:
  clusterIP: None
  selector:
    app: web
  ports:
    - port: 80
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: web-headless
  replicas: 3
  podManagementPolicy: OrderedReady
  selector:
    matchLabels:
      app: web
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80
          volumeMounts:
            - name: data
              mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 1Gi
      
        

          
          
          
          terminal
        

        # List StatefulSet pods (ordered names: web-0, web-1, web-2)
kubectl get pods -l app=web

# Scale a StatefulSet
kubectl scale statefulset web --replicas=5

# Canary update: only update pod web-2 and above
kubectl patch statefulset web -p '{"spec":{"updateStrategy":{"rollingUpdate":{"partition":2}}}}'

# Rollout the update to all pods
kubectl patch statefulset web -p '{"spec":{"updateStrategy":{"rollingUpdate":{"partition":0}}}}'

DaemonSet

A DaemonSet ensures one pod runs on every node (or a subset of nodes). Typical use cases include log collectors, monitoring agents, and network plugins. When a new node joins the cluster, the DaemonSet controller automatically creates a pod on it.

Key Spec Fields

FieldTypeDescription
selectorLabelSelectorRequired. Must match .spec.template.metadata.labels.
templatePodTemplateSpecPod template. Each node gets one pod.
updateStrategy.typestringRollingUpdate (default) or OnDelete.
updateStrategy.rollingUpdate.maxUnavailableint or %Max pods that can be unavailable during update. Default 1.
updateStrategy.rollingUpdate.maxSurgeint or %Max extra pods during update. Default 0.
minReadySecondsintegerSeconds a pod must be ready before considered available.
revisionHistoryLimitintegerNumber of old ControllerRevisions to keep. Default 10.
Pod naming

DaemonSet pod names are generated using a deterministic hash of the template, encoded with the Kubernetes-standard alphabet bcdfghjklmnpqrstvwxz2456789. This ensures consistent naming across controller restarts.

Node Selection

Use nodeSelector, affinity, or tolerations in the pod template to target specific nodes: 

        

          
          
          
          daemonset.yaml
        

        apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: node-exporter
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      tolerations:
        - operator: Exists    # Run on all nodes including masters
      hostNetwork: true
      containers:
        - name: exporter
          image: prom/node-exporter:v1.8.0
          ports:
            - containerPort: 9100
              hostPort: 9100
      
        

          
          
          
          terminal
        

        # List DaemonSets
kubectl get daemonsets -A

# Check desired/current/ready counts
kubectl get ds node-exporter -n kube-system

# Rollout status
kubectl rollout status daemonset/node-exporter -n kube-system

Job

A Job creates one or more pods and ensures a specified number of them run to successful completion. Jobs are for batch processing, data migrations, one-off tasks, and anything that should run to completion rather than continuously.

Key Spec Fields

FieldTypeDescription
completionsintegerNumber of successful completions needed. Default 1. Set to null for "any completion" mode.
parallelismintegerMax pods running concurrently. Default 1.
backoffLimitintegerNumber of retries before marking the job failed. Default 6.
activeDeadlineSecondsintegerMax lifetime of the job in seconds. Overrides backoffLimit.
ttlSecondsAfterFinishedintegerAuto-delete the job this many seconds after completion. Null = never delete.
completionModestringNonIndexed (default) or Indexed. Indexed mode sets JOB_COMPLETION_INDEX env var (0 to completions-1).
suspendbooleanSuspend the job (delete active pods, do not create new ones).
backoffLimitPerIndexintegerPer-index retry limit (Indexed mode only).
maxFailedIndexesintegerMax failed indexes before the whole job fails (Indexed mode only).

Completion Patterns

PatterncompletionsparallelismBehavior
Single pod11One pod runs to completion
Fixed completionsNMN pods must succeed, up to M at a time
Work queuenullMPods coordinate via an external queue; first success completes the job
IndexedNMEach pod gets a unique index; all N indexes must succeed

Example

        

          
          
          
          job.yaml
        

        apiVersion: batch/v1
kind: Job
metadata:
  name: data-migration
spec:
  completions: 10
  parallelism: 3
  backoffLimit: 4
  activeDeadlineSeconds: 600
  ttlSecondsAfterFinished: 3600
  completionMode: Indexed
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: myapp:migrate
          env:
            - name: BATCH_INDEX
              value: "$(JOB_COMPLETION_INDEX)"
      
        

          
          
          
          terminal
        

        # Create a quick job
kubectl create job pi --image=perl:5.34 -- perl -Mbignum=bpi -wle "print bpi(2000)"

# Watch job progress
kubectl get jobs -w

# View completed/failed pod count
kubectl describe job data-migration

# View logs from a job pod
kubectl logs job/data-migration

# Delete a job and its pods
kubectl delete job data-migration

CronJob

A CronJob creates Jobs on a time-based schedule. It uses standard cron syntax with five fields: minute, hour, day-of-month, month, day-of-week.

Key Spec Fields

FieldTypeDescription
schedulestringRequired. Cron expression, e.g., "*/5 * * * *" (every 5 minutes).
timeZonestringIANA timezone, e.g., "America/New_York". Default is controller-manager timezone.
concurrencyPolicystringAllow (default, concurrent jobs OK), Forbid (skip if previous still running), Replace (kill previous, start new).
successfulJobsHistoryLimitintegerNumber of successful jobs to keep. Default 3.
failedJobsHistoryLimitintegerNumber of failed jobs to keep. Default 1.
startingDeadlineSecondsintegerDeadline for starting the job if it missed its scheduled time. If more than 100 schedules are missed, the CronJob stops.
suspendbooleanSuspend the CronJob (no new jobs will be created).
jobTemplateJobTemplateSpecTemplate for jobs to create.

Example

        

          
          
          
          cronjob.yaml
        

        apiVersion: batch/v1
kind: CronJob
metadata:
  name: cleanup
spec:
  schedule: "0 2 * * *"        # Daily at 2am
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  startingDeadlineSeconds: 300
  jobTemplate:
    spec:
      backoffLimit: 2
      ttlSecondsAfterFinished: 86400
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: cleanup
              image: myapp:cleanup
              command: ["/bin/sh", "-c", "cleanup.sh"]
      
        

          
          
          
          terminal
        

        # Create a CronJob
kubectl create cronjob hello --image=busybox --schedule="*/1 * * * *" -- echo "Hello"

# List CronJobs
kubectl get cronjobs

# Trigger a job manually from a CronJob
kubectl create job cleanup-manual --from=cronjob/cleanup

# Suspend a CronJob
kubectl patch cronjob cleanup -p '{"spec":{"suspend":true}}'

# View scheduled jobs
kubectl describe cronjob cleanup

ReplicationController (Legacy)

ReplicationController is the predecessor to ReplicaSet. It functions the same way — maintaining a specified number of pod replicas — but uses equality-based selectors only (no matchExpressions). Use Deployments and ReplicaSets for new workloads.

Key Spec Fields

FieldTypeDescription
replicasintegerDesired number of replicas.
selectormapEquality-based label selector (key=value pairs only).
templatePodTemplateSpecPod template for creating replicas.
 
        

          
          
          
          rc.yaml
        

        apiVersion: v1
kind: ReplicationController
metadata:
  name: nginx-rc
spec:
  replicas: 3
  selector:
    app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
Deprecated

ReplicationController is supported for backward compatibility but considered deprecated. Use a Deployment with a ReplicaSet instead, which supports set-based selectors, rolling updates, and rollback.

Workload Comparison

Resource API Group Use Case Manages
Pod v1 Single instance, debugging, testing Containers directly
Deployment apps/v1 Stateless apps with rolling updates ReplicaSets → Pods
ReplicaSet apps/v1 Maintain replica count (usually via Deployment) Pods
StatefulSet apps/v1 Stateful apps needing stable identity and storage Pods + PVCs
DaemonSet apps/v1 One pod per node (agents, collectors) Pods
Job batch/v1 Run-to-completion batch tasks Pods
CronJob batch/v1 Scheduled recurring tasks Jobs → Pods
ReplicationController v1 Legacy replica management Pods

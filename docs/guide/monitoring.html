Monitoring & Observability

Health checks, Prometheus metrics, structured logging, and tracing for every Rusternetes component.

Health Endpoints

The API server exposes three health endpoints that return 200 OK when the component is healthy, matching the standard Kubernetes health check contract.

Endpoint Purpose Response
/healthz General health check (legacy) 200 OK with body ok
/livez Liveness probe — is the process alive? 200 OK with body ok
/readyz Readiness probe — is the process ready to serve? 200 OK with body ok 
        

          
          
          
          terminal
        

        # Check API server health
curl -k https://localhost:6443/healthz
# ok

curl -k https://localhost:6443/livez
# ok

curl -k https://localhost:6443/readyz
# ok
Docker health checks

The docker-compose.yml uses these endpoints for container health checks. Run docker compose ps to see the health status of each service.

Prometheus Metrics

Each component exposes a /metrics endpoint in Prometheus exposition format. These endpoints return counters, gauges, and histograms for request latency, queue depth, reconciliation cycles, and more.

Component Port Metrics Endpoint Additional Endpoints
API Server 6443 (main port) /metrics —
Kubelet 10250 (configurable) /metrics /configz
Scheduler 8081 /metrics —
Controller Manager 8080 /metrics — 
        

          
          
          
          terminal
        

        # Scrape API server metrics
curl -k https://localhost:6443/metrics

# Scrape kubelet metrics
curl -k https://localhost:10250/metrics

# Scrape kubelet config
curl -k https://localhost:10250/configz

# Scrape scheduler metrics
curl http://localhost:8081/metrics
Kubelet port

The kubelet metrics port defaults to 10250 but can be changed with the --port flag or KUBELET_PORT environment variable.

Tracing

Rusternetes uses the tracing and tracing-subscriber crates for structured logging and distributed tracing. OpenTelemetry integration is available via tracing-opentelemetry for exporting spans to Jaeger, Zipkin, or any OTLP-compatible collector. 

        

          
          
          
          terminal
        

        # Enable OpenTelemetry export to a local collector
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317
export OTEL_SERVICE_NAME=rusternetes-api-server

Logging

All Rusternetes components use Rust's tracing framework for structured logging. Log levels and component-specific filtering are controlled via the RUST_LOG environment variable or the --log-level CLI flag.

Log levels

Level Description
error Fatal or critical failures only
warn Warnings and unexpected conditions
info Normal operational messages (default)
debug Detailed diagnostic information
trace Very verbose, per-request/per-field logging

Setting log levels

        

          
          
          
          terminal
        

        # Global log level
export RUST_LOG=debug

# Component-specific logging
export RUST_LOG=rusternetes_api_server=debug

# Multiple components at different levels
export RUST_LOG=rusternetes_api_server=debug,rusternetes_scheduler=info,rusternetes_kubelet=trace

# Via CLI flag
./rusternetes-api-server --log-level debug
RUST_LOG syntax

The RUST_LOG variable uses the tracing-subscriber filter syntax. Crate names use underscores (e.g., rusternetes_api_server), not hyphens. You can also filter by module path: rusternetes_api_server::handlers=trace.

Viewing logs in Docker Compose

        

          
          
          
          terminal
        

        # Follow logs from all services
docker compose logs -f

# Follow logs from a single component
docker compose logs -f api-server
docker compose logs -f kubelet
docker compose logs -f scheduler
docker compose logs -f controller-manager
docker compose logs -f kube-proxy

# Show last 100 lines from the API server
docker compose logs --tail 100 api-server

# Show logs with timestamps
docker compose logs -f -t api-server

Enabling debug logging in Docker Compose

To enable debug-level logging for a specific service, set the RUST_LOG environment variable in docker-compose.yml or via an override: 

        

          
          
          
          docker-compose.override.yml
        

        services:
  api-server:
    environment:
      - RUST_LOG=rusternetes_api_server=debug
  scheduler:
    environment:
      - RUST_LOG=rusternetes_scheduler=debug
Performance impact

Running with trace or debug log levels in production generates significant output and can impact performance. Use info or warn for production clusters, and only enable debug logging temporarily for troubleshooting.

Monitoring Checklist

  • Verify all services are healthy: docker compose ps
  • Check API server readiness: curl -k https://localhost:6443/readyz
  • Verify nodes are reporting: kubectl get nodes
  • Check pod scheduling: kubectl get pods -A
  • Monitor component logs: docker compose logs -f
  • Scrape Prometheus metrics for dashboards and alerting

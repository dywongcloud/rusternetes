Security

Rusternetes implements Kubernetes security features including TLS/mTLS, RBAC authorization, ServiceAccount tokens (RS256 JWT), admission webhooks, ValidatingAdmissionPolicy with CEL, ResourceQuota, LimitRange, SecurityContext, and Secret management.

TLS / mTLS

The API server requires TLS for all connections. The bootstrap script generates self-signed certificates with 10-year validity. Certificates are stored in .rusternetes/certs/.

Certificate Requirements

RequirementDetails
SANsMust include localhost, 127.0.0.1, kubernetes, kubernetes.default, kubernetes.default.svc, kubernetes.default.svc.cluster.local, and Docker bridge IPs (172.18.0.2-5).
ALPNTLS ServerConfig must set alpn_protocols = [b"h2", b"http/1.1"]. Without ALPN, Go's client-go cannot negotiate HTTP/2 and watches fail with "context canceled".
Validity10-year validity for self-signed certs. Customize with --cert-validity-days.
Key typeRSA 2048-bit or ECDSA P-256.
HTTP/2 ALPN is required

Go's client-go library requires ALPN negotiation to select HTTP/2. If the TLS config does not advertise h2, watches will fail with "context canceled" errors because the client falls back to HTTP/1.1 without proper framing.

Generating Custom Certificates

        

          
          
          
          terminal
        

        # Generate certs (done automatically by bootstrap)
bash scripts/generate-certs.sh

# Custom SANs for non-standard deployments
openssl req -x509 -newkey rsa:2048 -nodes \
  -keyout server.key -out server.crt \
  -days 3650 \
  -subj "/CN=kube-apiserver" \
  -addext "subjectAltName=DNS:kubernetes,DNS:kubernetes.default,\
DNS:kubernetes.default.svc,DNS:kubernetes.default.svc.cluster.local,\
IP:127.0.0.1,IP:10.96.0.1,IP:172.18.0.2,IP:172.18.0.3"

Using Custom Certificates

        

          
          
          
          terminal
        

        # API server flags
rusternetes-api-server \
  --tls-cert-file=/path/to/server.crt \
  --tls-private-key-file=/path/to/server.key \
  --client-ca-file=/path/to/ca.crt

RBAC Authorization

Role-Based Access Control (RBAC) grants permissions to users, groups, and service accounts. RBAC uses four resource types: Role, ClusterRole, RoleBinding, and ClusterRoleBinding.

Role & ClusterRole

A Role grants permissions within a single namespace. A ClusterRole grants cluster-wide permissions or can be bound to a namespace via RoleBinding.

Rule FieldDescription
apiGroupsAPI groups ("" = core, "apps", "batch", etc.).
resourcesResource types ("pods", "deployments", "secrets", etc.).
verbsAllowed operations: get, list, watch, create, update, patch, delete, deletecollection.
resourceNamesOptional. Restrict to specific resource instances by name.
nonResourceURLsClusterRole only. Non-resource URLs like /healthz, /api.
 
        

          
          
          
          role.yaml
        

        # Namespace-scoped Role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/exec"]
    verbs: ["create"]
---
# Cluster-scoped ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: secret-reader
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get", "list", "watch"]
  - nonResourceURLs: ["/healthz", "/readyz"]
    verbs: ["get"]

AggregationRule

ClusterRoles can aggregate rules from other ClusterRoles using label selectors: 

        

          
          
          
          aggregated-clusterrole.yaml
        

        apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-view
aggregationRule:
  clusterRoleSelectors:
    - matchLabels:
        rbac.example.com/aggregate-to-monitoring: "true"
rules: []  # Rules are auto-filled by the controller

RoleBinding & ClusterRoleBinding

Bindings grant the permissions defined in a Role or ClusterRole to subjects (users, groups, or ServiceAccounts).

Subject KindDescription
UserUsername from the authentication layer (e.g., client certificate CN).
GroupGroup name (e.g., system:masters, system:authenticated).
ServiceAccountServiceAccount in a specific namespace. Format: system:serviceaccount:{ns}:{name}.
 
        

          
          
          
          bindings.yaml
        

        # RoleBinding - grants Role in a namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
  - kind: User
    name: jane
    apiGroup: rbac.authorization.k8s.io
  - kind: ServiceAccount
    name: my-app
    namespace: default
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
---
# ClusterRoleBinding - grants ClusterRole cluster-wide
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: admin-binding
subjects:
  - kind: Group
    name: system:masters
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-admin
  apiGroup: rbac.authorization.k8s.io
      
        

          
          
          
          terminal
        

        # Create RBAC resources via kubectl
kubectl create role pod-reader --verb=get,list,watch --resource=pods
kubectl create rolebinding read-pods --role=pod-reader --user=jane
kubectl create clusterrole node-reader --verb=get,list --resource=nodes
kubectl create clusterrolebinding read-nodes --clusterrole=node-reader --group=developers

# Check permissions
kubectl auth can-i get pods --namespace=default
kubectl auth can-i create deployments --all-namespaces
kubectl auth can-i "*" "*"  # Am I cluster admin?

# Who am I?
kubectl auth whoami

ServiceAccount Tokens

Rusternetes uses RS256 (RSA) JWT tokens for ServiceAccount authentication, compatible with OIDC discovery. Tokens are automatically projected into pods via the projected volume mechanism.

Token Details

PropertyValue
AlgorithmRS256 (RSA with SHA-256). Not HS256 (HMAC).
Signing key/etc/kubernetes/pki/sa.key (private key)
Verification key/etc/kubernetes/pki/sa.pub (public key)
Issuerhttps://kubernetes.default.svc.cluster.local
Audiencehttps://kubernetes.default.svc
Expiry3607 seconds (default, configurable)
OIDC discovery/.well-known/openid-configuration and /openid/v1/jwks

Auto-injected Files

Every pod automatically gets a projected volume mounted at /var/run/secrets/kubernetes.io/serviceaccount/ containing:

FileContent
tokenJWT token for the pod's ServiceAccount.
ca.crtCA certificate for verifying the API server's TLS cert.
namespaceThe pod's namespace.
 
        

          
          
          
          terminal
        

        # Create a ServiceAccount
kubectl create serviceaccount my-app

# Create a token manually
kubectl create token my-app --duration=1h

# Verify auto-injected token from inside a pod
kubectl exec my-pod -- cat /var/run/secrets/kubernetes.io/serviceaccount/token

Admission Webhooks

Admission webhooks intercept API requests after authentication and authorization but before persistence. Validating webhooks can accept or reject requests. Mutating webhooks can also modify the object.

ValidatingWebhookConfiguration

        

          
          
          
          validating-webhook.yaml
        

        apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: pod-policy
webhooks:
  - name: pod-policy.example.com
    admissionReviewVersions: ["v1"]
    sideEffects: None
    failurePolicy: Fail          # Fail or Ignore
    timeoutSeconds: 30
    clientConfig:
      service:
        name: webhook-service
        namespace: kube-system
        path: /validate
        port: 443
      caBundle: <base64-encoded-CA>
    rules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        resources: ["pods"]
        operations: ["CREATE", "UPDATE"]
    namespaceSelector:
      matchExpressions:
        - key: environment
          operator: In
          values: ["production"]

MutatingWebhookConfiguration

        

          
          
          
          mutating-webhook.yaml
        

        apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: sidecar-injector
webhooks:
  - name: sidecar.example.com
    admissionReviewVersions: ["v1"]
    sideEffects: None
    failurePolicy: Ignore
    reinvocationPolicy: IfNeeded
    clientConfig:
      service:
        name: sidecar-injector
        namespace: kube-system
        path: /mutate
      caBundle: <base64-encoded-CA>
    rules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        resources: ["pods"]
        operations: ["CREATE"]

AdmissionReview

Webhooks receive an AdmissionReview v1 request and must return an AdmissionReview response. For mutations, the response includes a JSON Patch (RFC 6902) in the patchType: JSONPatch field.

Webhook denial messages

When a webhook denies a request, the API server uses status.reason from the webhook response. If message is empty, the API server falls back to the reason field for the user-facing error message.

ValidatingAdmissionPolicy (CEL)

ValidatingAdmissionPolicy provides declarative, in-process validation using CEL (Common Expression Language) — no webhook server needed. 

        

          
          
          
          vap.yaml
        

        # Policy definition
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy
metadata:
  name: require-labels
spec:
  failurePolicy: Fail
  matchConstraints:
    resourceRules:
      - apiGroups: ["apps"]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["deployments"]
  validations:
    - expression: "has(object.metadata.labels) && 'team' in object.metadata.labels"
      message: "All deployments must have a 'team' label."
    - expression: "object.spec.replicas <= 10"
      message: "Replica count must not exceed 10."
---
# Binding (activates the policy)
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicyBinding
metadata:
  name: require-labels-binding
spec:
  policyName: require-labels
  validationActions: ["Deny"]
  matchResources:
    namespaceSelector:
      matchLabels:
        env: production

Built-in Admission Controllers

ResourceQuota

Enforces aggregate resource limits per namespace. Tracks CPU, memory, ephemeral-storage, pod count, and other resource totals.

ScopeDescription
TerminatingPods with activeDeadlineSeconds set.
NotTerminatingPods without activeDeadlineSeconds.
BestEffortPods with no resource requests or limits.
NotBestEffortPods with at least one resource request or limit.
 
        

          
          
          
          resourcequota.yaml
        

        apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: default
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    requests.ephemeral-storage: 50Gi
    limits.ephemeral-storage: 100Gi
    pods: "20"
    services: "10"
    secrets: "50"
    configmaps: "50"
    persistentvolumeclaims: "10"

LimitRange

Sets default, min, and max resource limits for containers in a namespace. 

        

          
          
          
          limitrange.yaml
        

        apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: default
spec:
  limits:
    - type: Container
      default:
        cpu: 500m
        memory: 256Mi
      defaultRequest:
        cpu: 100m
        memory: 128Mi
      min:
        cpu: 50m
        memory: 64Mi
      max:
        cpu: "2"
        memory: 2Gi
    - type: Pod
      max:
        cpu: "4"
        memory: 4Gi
    - type: PersistentVolumeClaim
      min:
        storage: 1Gi
      max:
        storage: 100Gi

ServiceAccount Injection

The ServiceAccount admission controller automatically adds a projected volume with the ServiceAccount token, CA certificate, and namespace to every pod that specifies a serviceAccountName (or uses the default).

SecurityContext

Security settings can be specified at both the pod level and the individual container level. Container-level settings override pod-level settings.

Pod-level SecurityContext

FieldDescription
runAsUserUID to run all containers as.
runAsGroupPrimary GID for all containers.
runAsNonRootRequire containers to run as non-root. Pod is rejected if image runs as root.
fsGroupGID applied to all volume mounts. Files created are owned by this GID.
supplementalGroupsAdditional GIDs for processes.
sysctlsKernel parameters to set (e.g., net.ipv4.ip_forward).
seccompProfileSeccomp profile: RuntimeDefault, Unconfined, or Localhost.
appArmorProfileAppArmor profile: RuntimeDefault, Unconfined, or Localhost.

Container-level SecurityContext

FieldDescription
privilegedRun the container in privileged mode (full host access).
capabilities.addLinux capabilities to add (e.g., NET_ADMIN, SYS_TIME).
capabilities.dropLinux capabilities to drop (e.g., ALL).
readOnlyRootFilesystemMount the container's root filesystem as read-only.
allowPrivilegeEscalationControls whether a process can gain more privileges than its parent.
procMountDefault or Unmasked. Controls /proc mount masking.
runAsUserUID override for this container.
runAsGroupGID override for this container.
runAsNonRootOverride for this container.
 
        

          
          
          
          securitycontext.yaml
        

        apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: myapp:latest
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
          add: ["NET_BIND_SERVICE"]
      volumeMounts:
        - name: tmp
          mountPath: /tmp
  volumes:
    - name: tmp
      emptyDir: {}

Secrets

Secrets store sensitive data like passwords, tokens, and TLS certificates. Data is base64-encoded in the API but not encrypted at rest (unless storage-level encryption is configured).

Secret Types

TypeDescriptionRequired Keys
OpaqueDefault. Arbitrary key-value data.None
kubernetes.io/tlsTLS certificate and key pair.tls.crt, tls.key
kubernetes.io/dockerconfigjsonDocker registry credentials..dockerconfigjson
kubernetes.io/service-account-tokenLegacy ServiceAccount token. Auto-populated by controller.Annotations: kubernetes.io/service-account.name
kubernetes.io/basic-authBasic authentication credentials.username, password
kubernetes.io/ssh-authSSH authentication credentials.ssh-privatekey
 
        

          
          
          
          terminal
        

        # Create secrets
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password=secret123

kubectl create secret tls app-tls \
  --cert=server.crt \
  --key=server.key

kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=user \
  --docker-password=pass

# View secret (base64 encoded)
kubectl get secret db-creds -o yaml

# Decode a secret value
kubectl get secret db-creds -o jsonpath='{.data.password}' | base64 -d

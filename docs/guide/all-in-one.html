All-in-One Binary

Run the entire Kubernetes control plane — API server, scheduler, controller manager, kubelet, and kube-proxy — in a single Rust process with embedded SQLite storage. No Docker Compose, no etcd, no external dependencies beyond Docker for running pod containers.

NOTE

The all-in-one binary still requires Docker to be running on the host machine. The binary itself does not run in Docker, but it uses the Docker API (via bollard) to create and manage pod containers.

Quick Start

        

          
          
          
          terminal
        

        
# Build and run with defaults (SQLite, no TLS, port 6443)
cargo run --release --bin rusternetes

# Or if you have the binary already built:
./target/release/rusternetes

That's it. The API server is now listening on 0.0.0.0:6443. You can immediately use kubectl to interact with it: 

        

          
          
          
          terminal
        

        
# Point kubectl at the local API server (no TLS)
kubectl --server=http://localhost:6443 get nodes

# Or set up a kubeconfig
kubectl config set-cluster rusternetes --server=http://localhost:6443
kubectl config set-context rusternetes --cluster=rusternetes
kubectl config use-context rusternetes

CLI Flags Reference

Every flag has a sensible default. The binary runs with zero flags for the simplest development experience.

Storage

Flag Type Default Description
--storage-backend String "sqlite" Storage backend to use. Accepts sqlite (embedded, no dependencies) or etcd (requires a running etcd server).
--data-dir String "./data/rusternetes.db" Path to the SQLite database file. The directory is created automatically if it does not exist. Only used when --storage-backend=sqlite.
--etcd-servers String "http://localhost:2379" Comma-separated list of etcd endpoints. Only used when --storage-backend=etcd.

Networking

Flag Type Default Description
--bind-address String "0.0.0.0:6443" Address and port for the API server to listen on. Use 127.0.0.1:6443 to restrict to localhost only.
--cluster-dns String "10.96.0.10" ClusterIP address for the DNS service. Injected into pod /etc/resolv.conf by the kubelet.
--network String "rusternetes-network" Docker network name for pod containers. Created automatically if it does not exist.

TLS

Flag Type Default Description
--tls bool false Enable TLS on the API server. When enabled, --tls-cert-file and --tls-key-file must be provided.
--tls-cert-file Option<String> None Path to the PEM-encoded TLS certificate file. Required when --tls is enabled.
--tls-key-file Option<String> None Path to the PEM-encoded TLS private key file. Required when --tls is enabled.
--tls-san String "localhost,127.0.0.1" Comma-separated Subject Alternative Names for the TLS certificate. Add hostnames and IPs that clients will use to connect.

Node

Flag Type Default Description
--node-name String "node-1" Name for the embedded kubelet's node registration. Appears in kubectl get nodes.
--volume-dir String "./data/volumes" Directory where pod volumes (emptyDir, hostPath, etc.) are stored on the host filesystem.

Component Intervals

Flag Type Default Description
--sync-interval u64 5 Controller manager reconciliation interval in seconds. Each of the 31 controllers runs a reconcile loop at this interval.
--scheduler-interval u64 2 Scheduler check interval in seconds. How often the scheduler looks for unscheduled pods.
--kubelet-sync-interval u64 3 Kubelet sync interval in seconds. How often the kubelet reconciles desired vs. actual pod state.
--proxy-sync-interval u64 1 Kube-proxy sync interval in seconds. How often iptables rules are regenerated from Services and Endpoints.

Auth & Proxy

Flag Type Default Description
--skip-auth bool true Skip RBAC authentication and authorization. Default is true for development convenience. Set to false for production use with ServiceAccount tokens.
--disable-proxy bool false Disable the embedded kube-proxy. Useful when running without CAP_NET_ADMIN or when iptables is not available on the host.
--log-level String "info" Log verbosity level. Accepts trace, debug, info, warn, or error. Uses the RUST_LOG env var format internally.

Usage Examples

Basic (defaults)

        

          
          
          
          terminal
        

        
# Simplest invocation: SQLite, no TLS, port 6443, auth disabled
./rusternetes

With TLS

        

          
          
          
          terminal
        

        
# Generate certs first (or bring your own)
bash scripts/generate-certs.sh

# Run with TLS enabled
./rusternetes \
  --tls \
  --tls-cert-file=.rusternetes/certs/server.crt \
  --tls-key-file=.rusternetes/certs/server.key

With custom SANs

        

          
          
          
          terminal
        

        
# Add extra SANs for remote access
./rusternetes \
  --tls \
  --tls-cert-file=certs/server.crt \
  --tls-key-file=certs/server.key \
  --tls-san="localhost,127.0.0.1,192.168.1.100,k8s.example.com"

With etcd instead of SQLite

        

          
          
          
          terminal
        

        
# Start etcd first
etcd --listen-client-urls=http://0.0.0.0:2379 \
     --advertise-client-urls=http://localhost:2379

# Point the binary at etcd
./rusternetes \
  --storage-backend=etcd \
  --etcd-servers=http://localhost:2379

Full production example

        

          
          
          
          terminal
        

        
./rusternetes \
  --storage-backend=sqlite \
  --data-dir=/var/lib/rusternetes/state.db \
  --volume-dir=/var/lib/rusternetes/volumes \
  --bind-address=0.0.0.0:6443 \
  --node-name=edge-node-01 \
  --tls \
  --tls-cert-file=/etc/rusternetes/tls/server.crt \
  --tls-key-file=/etc/rusternetes/tls/server.key \
  --tls-san="edge-node-01.example.com,10.0.0.50" \
  --skip-auth=false \
  --log-level=info \
  --sync-interval=10 \
  --scheduler-interval=5 \
  --kubelet-sync-interval=5 \
  --proxy-sync-interval=2

Backing Up & Inspecting State

When using the SQLite backend, all cluster state — pods, deployments, services, secrets, configmaps, and everything else — lives in a single database file. This makes backup and inspection trivial.

Backup

        

          
          
          
          terminal
        

        
# Hot backup: SQLite supports concurrent reads
cp ./data/rusternetes.db ./backups/rusternetes-$(date +%Y%m%d-%H%M%S).db

# Or use sqlite3's .backup for a consistent snapshot
sqlite3 ./data/rusternetes.db ".backup ./backups/rusternetes-backup.db"

Inspect state

        

          
          
          
          terminal
        

        
# Open the database
sqlite3 ./data/rusternetes.db

# List all stored keys (resource paths)
SELECT key FROM kv ORDER BY key;

# View a specific resource
SELECT value FROM kv WHERE key = '/registry/pods/default/nginx';

# Count resources by type
SELECT
  substr(key, 1, instr(substr(key, 11), '/') + 10) AS resource_type,
  count(*) AS count
FROM kv
WHERE key LIKE '/registry/%'
GROUP BY resource_type
ORDER BY count DESC;

# Check database size
SELECT page_count * page_size AS size_bytes FROM pragma_page_count(), pragma_page_size();

Restore from backup

        

          
          
          
          terminal
        

        
# Stop the binary first
kill $(pgrep rusternetes)

# Replace the database file
cp ./backups/rusternetes-backup.db ./data/rusternetes.db

# Restart
./rusternetes
WARNING

Do not replace the database file while the binary is running. SQLite uses file-level locking, and replacing the file mid-operation can corrupt state. Always stop the process first.

Use Cases

Edge Computing

Deploy the all-in-one binary on edge nodes with limited resources. A single ARM64 or x86 device with 512 MB RAM and Docker can run a full Kubernetes API with pod scheduling. State persists across reboots in the SQLite file. 

        

          
          
          
          terminal
        

        
# Edge deployment with minimal intervals
./rusternetes \
  --node-name=edge-gateway \
  --data-dir=/opt/rusternetes/state.db \
  --volume-dir=/opt/rusternetes/volumes \
  --sync-interval=30 \
  --scheduler-interval=10 \
  --disable-proxy

CI/CD Pipelines

Spin up a Kubernetes API in your CI pipeline to test Helm charts, operators, or client-go code without needing kind, minikube, or a cloud cluster. The binary starts in under a second. 

        

          
          
          
          ci-pipeline.yaml
        

        
steps:
  - name: Start Kubernetes
    run: |
      ./rusternetes --log-level=warn &
      sleep 2
      kubectl --server=http://localhost:6443 cluster-info

  - name: Run integration tests
    run: |
      helm install my-app ./charts/my-app \
        --set image.tag=$CI_SHA
      kubectl wait --for=condition=available deployment/my-app
      go test ./integration/...

IoT & Air-Gapped Environments

In air-gapped or disconnected environments, the binary plus pre-pulled container images are all you need. No internet access is required at runtime. 

        

          
          
          
          terminal
        

        
# On a connected machine: save images
docker save nginx:alpine busybox:latest | gzip > images.tar.gz

# On the air-gapped device: load images
docker load < images.tar.gz

# Run rusternetes (no internet needed)
./rusternetes --node-name=iot-hub

Developer Sandbox

Run the binary during development to get a local Kubernetes API without Docker Compose setup. Changes to the codebase can be tested with cargo run immediately. 

        

          
          
          
          terminal
        

        
# Development workflow
cargo run --bin rusternetes -- --log-level=debug

# In another terminal
kubectl apply -f my-deployment.yaml
kubectl get pods -w

Embedded Sandbox (OpenShell)

Embed rusternetes as a library or spawn it as a subprocess inside a larger application. This is how OpenShell provides each user session with an isolated Kubernetes environment. Each instance gets its own SQLite file, providing complete tenant isolation. 

        

          
          
          
          terminal
        

        
# Per-session Kubernetes sandbox
./rusternetes \
  --data-dir=/tmp/sessions/user-abc123/state.db \
  --volume-dir=/tmp/sessions/user-abc123/volumes \
  --bind-address=127.0.0.1:0 \
  --node-name=sandbox-abc123 \
  --disable-proxy

How It Works

The rusternetes binary (in crates/rusternetes/) creates a shared StorageBackend and spawns each component as a concurrent tokio task:

  1. Storage — Creates a StorageBackend::Sqlite or StorageBackend::Etcd based on --storage-backend.
  2. API Server — Builds the Axum router with all 76 handlers and starts listening on --bind-address.
  3. Controller Manager — Spawns all 31 controllers as concurrent tasks, sharing the same storage Arc.
  4. Scheduler — Watches for unscheduled pods and assigns them to the node.
  5. Kubelet — Manages pod containers via the Docker API (bollard).
  6. Kube-Proxy — (unless --disable-proxy) Syncs iptables rules for Service routing.

All components share the same in-process storage instance. There is zero network overhead for storage operations — reads and writes go directly to SQLite.

TIP

Because all components share one process, you get a fully consistent view of state with no replication lag. Watch events are delivered in-process, which makes the all-in-one binary the fastest deployment mode for watch-heavy workloads.

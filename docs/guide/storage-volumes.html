Storage & Volumes

Rusternetes supports 12 volume types for mounting data into pod containers. This page covers every volume type, the PersistentVolume / PersistentVolumeClaim lifecycle, StorageClasses, dynamic provisioning, volume snapshots, and online expansion.

Volume Types

1. emptyDir

A temporary directory created when a pod is assigned to a node and deleted when the pod is removed. All containers in the pod can read and write to it. Useful for scratch space, caches, and sharing data between containers. 

        

          
          
          
          emptydir.yaml
        

        volumes:
  - name: cache
    emptyDir:
      medium: ""               # "" = disk (default), "Memory" = tmpfs
      sizeLimit: 256Mi         # Optional size limit
containers:
  - name: app
    volumeMounts:
      - name: cache
        mountPath: /tmp/cache
POSIX permissions

emptyDir volumes use a container-local path on the kubelet host, ensuring proper POSIX permission support for fsGroup, chmod, and file ownership changes.

2. hostPath

Mounts a file or directory from the host node's filesystem. Use with caution — pods become tied to a specific node and can access sensitive host data.

typeDescription
""No check (default). Path may or may not exist.
DirectoryMust be an existing directory.
DirectoryOrCreateCreate directory if missing (0755, same owner as kubelet).
FileMust be an existing file.
FileOrCreateCreate file if missing (0644, same owner as kubelet).
SocketMust be an existing UNIX socket.
CharDeviceMust be an existing character device.
BlockDeviceMust be an existing block device.
 
        

          
          
          
          hostpath.yaml
        

        volumes:
  - name: docker-sock
    hostPath:
      path: /var/run/docker.sock
      type: Socket
containers:
  - name: docker-cli
    volumeMounts:
      - name: docker-sock
        mountPath: /var/run/docker.sock

3. configMap

Mounts a ConfigMap's data as files in the container. Each key becomes a filename and its value becomes the file content. 

        

          
          
          
          configmap-volume.yaml
        

        volumes:
  - name: config
    configMap:
      name: app-config        # ConfigMap name
      defaultMode: 0644        # File permissions (default 0644)
      optional: false          # Fail pod if ConfigMap missing?
      items:                   # Optional: select specific keys
        - key: nginx.conf
          path: nginx.conf     # Filename in the mount
        - key: mime.types
          path: mime.types
containers:
  - name: nginx
    volumeMounts:
      - name: config
        mountPath: /etc/nginx/conf.d
        readOnly: true

4. secret

Mounts a Secret's data as files. Identical to configMap volumes but for sensitive data. Files are base64-decoded automatically. 

        

          
          
          
          secret-volume.yaml
        

        volumes:
  - name: tls
    secret:
      secretName: app-tls      # Secret name
      defaultMode: 0400         # Restrictive permissions
      optional: false
      items:
        - key: tls.crt
          path: cert.pem
        - key: tls.key
          path: key.pem
containers:
  - name: app
    volumeMounts:
      - name: tls
        mountPath: /etc/ssl/app
        readOnly: true

5. downwardAPI

Exposes pod and container metadata as files. Supports both fieldRef (pod metadata) and resourceFieldRef (container resource limits/requests).

fieldRef.fieldPathValue
metadata.namePod name
metadata.namespacePod namespace
metadata.uidPod UID
metadata.labelsAll labels, one per line as key="value"
metadata.annotationsAll annotations, one per line
spec.nodeNameNode the pod is running on
spec.serviceAccountNameServiceAccount name
status.hostIPNode IP address
status.podIPPod IP address
 
        

          
          
          
          downwardapi-volume.yaml
        

        volumes:
  - name: podinfo
    downwardAPI:
      items:
        - path: labels
          fieldRef:
            fieldPath: metadata.labels
        - path: cpu-limit
          resourceFieldRef:
            containerName: app
            resource: limits.cpu
containers:
  - name: app
    volumeMounts:
      - name: podinfo
        mountPath: /etc/podinfo

6. projected

Combines multiple volume sources into a single mount directory. Can project serviceAccountToken, configMap, secret, and downwardAPI sources together. This is the mechanism used for automatic ServiceAccount token injection. 

        

          
          
          
          projected-volume.yaml
        

        volumes:
  - name: kube-api-access
    projected:
      defaultMode: 0644
      sources:
        - serviceAccountToken:
            expirationSeconds: 3607
            path: token
        - configMap:
            name: kube-root-ca.crt
            items:
              - key: ca.crt
                path: ca.crt
        - downwardAPI:
            items:
              - path: namespace
                fieldRef:
                  fieldPath: metadata.namespace
        - secret:
            name: extra-creds
            items:
              - key: password
                path: password
containers:
  - name: app
    volumeMounts:
      - name: kube-api-access
        mountPath: /var/run/secrets/kubernetes.io/serviceaccount
        readOnly: true
Projected volume resync

When the kubelet resyncs projected volumes, it deletes files not in the expected file list. ServiceAccountToken files must be listed in the expected files or they will be removed on resync.

7. persistentVolumeClaim

References a PersistentVolumeClaim to mount a PersistentVolume into the pod. The PVC must be bound to a PV before the pod can start. 

        

          
          
          
          pvc-volume.yaml
        

        volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc        # PVC name in the same namespace
      readOnly: false
containers:
  - name: app
    volumeMounts:
      - name: data
        mountPath: /data

8. nfs

Mounts an NFS share directly. The NFS server must be accessible from the node. 

        

          
          
          
          nfs-volume.yaml
        

        volumes:
  - name: nfs-data
    nfs:
      server: 10.0.0.100       # NFS server address
      path: /exports/data       # Export path
      readOnly: false

9. iscsi

Mounts an iSCSI (SCSI over IP) block storage device. Provides access to enterprise SAN storage. 

        

          
          
          
          iscsi-volume.yaml
        

        volumes:
  - name: iscsi-data
    iscsi:
      targetPortal: 10.0.0.50:3260
      iqn: iqn.2025-01.com.example:storage
      lun: 0
      fsType: ext4
      readOnly: false

10. csi

Container Storage Interface — the standard plugin mechanism for external storage providers. CSI drivers run as separate pods and handle volume provisioning, attaching, and mounting. 

        

          
          
          
          csi-volume.yaml
        

        volumes:
  - name: csi-data
    csi:
      driver: ebs.csi.aws.com
      volumeHandle: vol-0abc123def456
      readOnly: false
      fsType: ext4
      volumeAttributes:
        storage.kubernetes.io/csiProvisionerIdentity: "1234"

11. ephemeral

An inline PVC that is created when the pod is scheduled and deleted when the pod is removed. Combines the simplicity of emptyDir with the flexibility of PVCs (StorageClass, access modes, size). 

        

          
          
          
          ephemeral-volume.yaml
        

        volumes:
  - name: scratch
    ephemeral:
      volumeClaimTemplate:
        metadata:
          labels:
            type: scratch
        spec:
          accessModes: ["ReadWriteOnce"]
          storageClassName: fast
          resources:
            requests:
              storage: 10Gi

12. image

Mounts the contents of an OCI image as a read-only volume. The image is pulled by the container runtime and its filesystem is mounted directly. 

        

          
          
          
          image-volume.yaml
        

        volumes:
  - name: static-assets
    image:
      reference: myregistry.com/assets:v1.2
      pullPolicy: IfNotPresent
containers:
  - name: web
    volumeMounts:
      - name: static-assets
        mountPath: /usr/share/nginx/html
        readOnly: true

Volume Type Summary

Volume TypePersistsSharedUse Case
emptyDirPod lifetimeSame podScratch space, inter-container data sharing
hostPathNode lifetimeSame nodeNode-level logs, Docker socket
configMapConfigMap lifetimeSame podApplication configuration files
secretSecret lifetimeSame podTLS certificates, credentials
downwardAPIPod lifetimeSame podPod metadata (labels, annotations, resources)
projectedVariesSame podCombined SA tokens + ConfigMaps + Secrets
persistentVolumeClaimPV lifetimeCross-podDatabases, stateful data
nfsNFS serverCross-podShared filesystem across pods

imageImage lifetimeRead-onlyStatic assets from container images

PersistentVolume & PersistentVolumeClaim

PersistentVolumes (PVs) are cluster-scoped storage resources. PersistentVolumeClaims (PVCs) are namespace-scoped requests for storage. The PV controller binds PVCs to matching PVs.

Lifecycle

PhasePV StatusDescription
ProvisioningAvailablePV created (statically or dynamically). Waiting for a matching PVC.
BindingBoundPVC matched and bound to this PV. Both reference each other.
UsingBoundPod is actively using the PVC. PV is mounted on a node.
ReclaimingReleasedPVC deleted. PV enters reclaim phase based on persistentVolumeReclaimPolicy.

Access Modes

ModeAbbreviationDescription
ReadWriteOnceRWOMount as read-write by a single node.
ReadOnlyManyROXMount as read-only by many nodes.
ReadWriteManyRWXMount as read-write by many nodes.
ReadWriteOncePodRWOPMount as read-write by a single pod.

Example

        

          
          
          
          pv-pvc.yaml
        

        # PersistentVolume
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-data
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:
    path: /mnt/data
---
# PersistentVolumeClaim
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 5Gi
      
        

          
          
          
          terminal
        

        # List PersistentVolumes
kubectl get pv

# List PersistentVolumeClaims
kubectl get pvc

# Describe binding status
kubectl describe pvc my-pvc

# Delete a PVC (PV reclaim policy determines what happens to data)
kubectl delete pvc my-pvc

StorageClass

A StorageClass defines a "class" of storage with a provisioner, parameters, and reclaim policy. PVCs that reference a StorageClass trigger dynamic provisioning of a PV.

Key Spec Fields

FieldTypeDescription
provisionerstringRequired. Name of the volume provisioner (e.g., kubernetes.io/aws-ebs, ebs.csi.aws.com).
reclaimPolicystringRetain (keep PV after PVC deletion) or Delete (delete PV and underlying storage). Default Delete.
allowVolumeExpansionbooleanAllow PVCs to request more storage than originally provisioned.
volumeBindingModestringImmediate (bind PVC immediately) or WaitForFirstConsumer (bind when pod is scheduled).
parametersmapProvisioner-specific key-value pairs (e.g., type: gp3, iopsPerGB: "50").
mountOptionsarrayMount options for PVs created by this class (e.g., ["nfsvers=4.1"]).
 
        

          
          
          
          storageclass.yaml
        

        apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com
reclaimPolicy: Delete
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer
parameters:
  type: gp3
  iopsPerGB: "50"
  encrypted: "true"

Dynamic Provisioning

The DynamicProvisionerController watches for unbound PVCs that reference a StorageClass. When a match is found, it automatically creates a PV with the requested capacity and binds it to the PVC. 

        

          
          
          
          architecture
        

        # Dynamic provisioning flow
1. User creates PVC with storageClassName: fast
2. DynamicProvisionerController detects unbound PVC
3. Controller creates PV matching PVC spec + StorageClass params
4. PV controller binds PVC <-> PV
5. Pod starts and mounts the volume
DEFAULT STORAGECLASS

Rusternetes bootstraps a default standard StorageClass on startup with the rusternetes.io/hostpath provisioner. PVCs that don't specify a StorageClass use this default automatically. You can manage StorageClasses and PVCs through the web console Storage page.

CSI Driver Setup

The Container Storage Interface (CSI) is the standard plugin mechanism for connecting external storage systems to Kubernetes. CSI drivers run as pods in the cluster and handle volume provisioning, attaching, and mounting.

How CSI Works

A CSI driver consists of:

  • Controller plugin — runs as a Deployment, handles volume creation/deletion
  • Node plugin — runs as a DaemonSet on every node, handles volume mount/unmount
  • CSIDriver resource — registered in the API, tells the kubelet how to interact with the driver
  • StorageClass — references the CSI driver as its provisioner

Installing a CSI Driver

Most CSI drivers are installed via kubectl manifests or Helm charts. The general pattern: 

        

          
          
          
          Terminal
        

        # Example: install a CSI driver
kubectl apply -f https://example.com/csi-driver/deploy.yaml

# Verify the CSIDriver resource was created
kubectl get csidrivers

# Create a StorageClass that uses the CSI driver
cat <<EOF | kubectl apply -f -
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: csi-storage
provisioner: csi.example.com
parameters:
  type: ssd
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
EOF

# Create a PVC using the CSI StorageClass
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-csi-volume
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: csi-storage
  resources:
    requests:
      storage: 10Gi
EOF

Supported CSI Operations

Operation Description Status
CreateVolume / DeleteVolumeProvision and remove volumesSupported
ControllerPublishVolumeAttach volume to a nodeSupported
NodeStageVolume / NodePublishVolumeMount volume in podSupported
VolumeAttachmentTrack volume attachment stateSupported
CSIDriver / CSINodeDriver discovery and capabilitiesSupported
Volume SnapshotsPoint-in-time volume copiesSupported
Volume ExpansionOnline resize of PVCsSupported

Built-in Provisioner

Without a CSI driver, Rusternetes includes a built-in hostpath provisioner (rusternetes.io/hostpath) that creates PVs backed by directories on the node’s filesystem. This is suitable for development and single-node clusters. For production, use a proper CSI driver that provides replicated storage.

Volume Snapshots

Volume snapshots provide point-in-time copies of PersistentVolumes.

Resources

ResourceScopeDescription
VolumeSnapshotClassClusterDefines the CSI driver and parameters for taking snapshots.
VolumeSnapshotNamespaceRequest to snapshot a PVC. References a VolumeSnapshotClass.
VolumeSnapshotContentClusterThe actual snapshot resource. Bound to a VolumeSnapshot.
 
        

          
          
          
          snapshot.yaml
        

        # VolumeSnapshotClass
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: csi-snapclass
driver: ebs.csi.aws.com
deletionPolicy: Delete
---
# VolumeSnapshot
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: my-snapshot
spec:
  volumeSnapshotClassName: csi-snapclass
  source:
    persistentVolumeClaimName: my-pvc

Restoring from Snapshot

        

          
          
          
          restore-pvc.yaml
        

        apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: restored-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: fast
  resources:
    requests:
      storage: 10Gi
  dataSource:
    name: my-snapshot
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io

Volume Expansion

PVCs can be expanded online if the StorageClass has allowVolumeExpansion: true. The PVC controller detects size increases and updates the PV capacity. 

        

          
          
          
          terminal
        

        # Expand a PVC from 5Gi to 20Gi
kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'

# Check expansion status
kubectl describe pvc my-pvc
# Look for condition: FileSystemResizePending or condition type Resizing
Online expansion

Most CSI drivers support online expansion (no pod restart needed). The filesystem resize happens transparently while the pod is running.

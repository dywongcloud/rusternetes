Docker Compose with SQLite (Rhino)

Run a multi-node Rusternetes cluster with SQLite as the storage backend instead of etcd. The Rhino server provides an etcd-compatible gRPC API backed by SQLite, giving you the same multi-node architecture with simpler operations and a smaller footprint.

What is Rhino?

Rhino is an etcd-compatible gRPC server that implements the etcd v3 API (KV, Watch, Lease) using a relational database backend. It supports:

  • SQLite — single-file embedded database, no external dependencies
  • PostgreSQL — for environments with existing PostgreSQL infrastructure
  • MySQL — for environments with existing MySQL infrastructure

From the Rusternetes components' perspective, Rhino is indistinguishable from etcd. The API server, scheduler, controller manager, kubelets, and kube-proxy all connect to Rhino using the standard etcd client library. No code changes are needed — you simply swap the service in your Docker Compose file.

NOTE

Rhino is designed for single-writer deployments. It does not support etcd's Raft consensus or multi-node replication. For high availability, use the etcd-based deployment.

Difference from etcd Mode

The only difference between the etcd and SQLite cluster deployments is the storage service. All other services — api-server, scheduler, controller-manager, kubelets, and kube-proxy — are identical.

Aspect etcd Mode SQLite Mode (Rhino)
Storage service etcd (quay.io/coreos/etcd) rhino (ghcr.io/calfonso/rhino)
Compose file docker-compose.yml docker-compose.sqlite.yml
State storage etcd data directory (bbolt) Single SQLite file
Memory usage ~300-500 MB (etcd) ~50-100 MB (rhino)
Backup method etcdctl snapshot save cp state.db backup.db
HA capable Yes (Raft consensus) No (single writer)
All other services Identical Identical

Rhino Configuration

The Rhino service in docker-compose.sqlite.yml is configured with these settings:

Setting Value Description
listen-address 0.0.0.0:2379 gRPC listen address. Matches etcd's client port so all components connect without changes.
endpoint sqlite:///data/rusternetes.db Database connection string. For SQLite, this is a file path. For PostgreSQL, use postgres://user:pass@host/db.
max-connections 20 Maximum number of database connections in the pool. SQLite supports concurrent reads but serializes writes. 
        

          
          
          
          docker-compose.sqlite.yml (rhino service)
        

        
services:
  rhino:
    image: ghcr.io/calfonso/rhino:latest
    command:
      - --listen-address=0.0.0.0:2379
      - --endpoint=sqlite:///data/rusternetes.db
      - --max-connections=20
    volumes:
      - rhino-data:/data
    networks:
      - rusternetes-net

  api-server:
    # Connects to rhino:2379 — same as etcd:2379
    environment:
      - ETCD_SERVERS=http://rhino:2379
    depends_on:
      - rhino
    # ... rest is identical to docker-compose.yml

Step-by-Step Setup

1. Set the volumes path

        

          
          
          
          terminal
        

        
export KUBELET_VOLUMES_PATH=$(pwd)/.rusternetes/volumes
mkdir -p $KUBELET_VOLUMES_PATH

2. Generate TLS certificates

        

          
          
          
          terminal
        

        
bash scripts/generate-certs.sh

3. Build Docker images

        

          
          
          
          terminal
        

        
# Build using the SQLite compose file
docker compose -f docker-compose.sqlite.yml build

4. Start the cluster

        

          
          
          
          terminal
        

        
docker compose -f docker-compose.sqlite.yml up -d

5. Bootstrap and verify

        

          
          
          
          terminal
        

        
# Bootstrap CoreDNS, services, and SA tokens
bash scripts/bootstrap-cluster.sh

# Verify
export KUBECONFIG=~/.kube/rusternetes-config
kubectl get nodes
kubectl get pods -A

Advantages

No etcd

etcd requires careful capacity planning, defragmentation, compaction monitoring, and backup procedures. With Rhino + SQLite, state management is as simple as managing a single file.

Simpler operations

  • Backup: cp state.db backup.db (no etcdctl snapshot required)
  • Restore: cp backup.db state.db and restart
  • Inspect: sqlite3 state.db "SELECT key FROM kv"
  • Size check: ls -lh state.db
  • No compaction: SQLite handles its own page management
  • No defragmentation: Use VACUUM if needed, but rarely necessary

Lighter resource footprint

etcd typically uses 300-500 MB of RAM and requires fast SSD storage for its write-ahead log. Rhino with SQLite uses 50-100 MB and works fine on any storage medium, including SD cards and network-attached storage.

Single file for all state

Every resource in the cluster — pods, services, secrets, configmaps, CRDs, leases, and everything else — lives in one SQLite file. You can copy this file to another machine and start a cluster with the exact same state.

Inspecting State with SQLite

        

          
          
          
          terminal
        

        
# Connect to the Rhino container's database
docker compose -f docker-compose.sqlite.yml exec rhino \
  sqlite3 /data/rusternetes.db

# Or copy the database locally and inspect
docker compose -f docker-compose.sqlite.yml cp rhino:/data/rusternetes.db ./state.db
sqlite3 ./state.db

# List all resources
.headers on
.mode column
SELECT key, length(value) AS bytes FROM kv ORDER BY key;

# Find all pods
SELECT key FROM kv WHERE key LIKE '/registry/pods/%';

# View a specific resource as JSON
SELECT value FROM kv WHERE key = '/registry/services/default/kubernetes';

# Database statistics
SELECT * FROM pragma_database_list;
SELECT page_count * page_size / 1024 / 1024 AS size_mb
FROM pragma_page_count(), pragma_page_size();

Switching Between etcd and SQLite

You can switch between deployment modes without changing any workload YAML. The API is identical — only the storage backend changes.

WARNING

State is not automatically migrated between etcd and SQLite. If you switch modes, you start with a fresh cluster. To migrate state, you would need to export all resources from one backend and re-apply them to the other using kubectl. 

        

          
          
          
          terminal
        

        
# Stop the etcd cluster
docker compose down

# Start with SQLite instead
docker compose -f docker-compose.sqlite.yml up -d

# Or switch back to etcd
docker compose -f docker-compose.sqlite.yml down
docker compose up -d

Troubleshooting

Symptom Cause Fix
Rhino fails to start Data volume permission issue Check that the rhino-data volume is writable. Try docker volume rm and recreate.
"database is locked" errors Too many concurrent writers Increase --max-connections or check for long-running transactions.
API server can't connect to storage Rhino not ready yet The api-server retries on startup. Check docker compose logs rhino for errors.
Watch events not delivered Rhino watch implementation issue Restart the rhino service. Check Rhino logs for panics or errors.
Database file growing large Historical revisions accumulating Rhino auto-compacts, but you can run sqlite3 state.db "VACUUM" to reclaim space.

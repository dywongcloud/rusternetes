Cluster Bootstrap

The bootstrap process creates the essential resources every Kubernetes cluster needs: namespaces, the kubernetes service, PriorityClasses, CoreDNS, and ServiceAccount tokens.

What bootstrap-cluster.sh Does

The scripts/bootstrap-cluster.sh script automates the full bootstrap sequence. Run it after your cluster components are up: 

        

          
          
          
          terminal
        

        bash scripts/bootstrap-cluster.sh

The script performs these steps in order:

1. Generate ServiceAccount Tokens

Calls scripts/generate-default-serviceaccounts.sh to create RS256 JWT tokens for the default ServiceAccount in each namespace. These tokens are signed with the SA private key at .rusternetes/certs/sa.key.

2. Apply ServiceAccounts and Secrets

Applies .rusternetes/default-serviceaccounts.yaml, which contains the ServiceAccount objects and their associated token Secrets for kube-system and default namespaces.

3. Clean Up Existing CoreDNS

Removes any existing CoreDNS container and etcd key to ensure a fresh deployment with the latest ServiceAccount token mounted.

4. Apply Bootstrap Resources

Applies bootstrap-cluster.yaml, which creates:

Resource Name Details
Namespace kube-system System components namespace
Namespace default Default namespace for user workloads
Service kubernetes ClusterIP 10.96.0.1, port 443 → 6443
PriorityClass system-node-critical Priority 2000001000 — node-level system pods
PriorityClass system-cluster-critical Priority 2000000000 — cluster-level system pods
Pod coredns CoreDNS 1.11.1 in kube-system
Service kube-dns ClusterIP 10.96.0.10, DNS ports 53 UDP/TCP
ConfigMap coredns Corefile configuration

5. Wait for CoreDNS

Polls the CoreDNS pod status for up to 30 seconds, waiting for it to reach Running phase. Reports final cluster state.

CoreDNS Configuration

The CoreDNS Corefile is stored in a ConfigMap and mounted into the CoreDNS pod. It configures cluster DNS resolution: 

        

          
          
          
          Corefile
        

        .:53 {
    errors
    health {
      lameduck 5s
    }
    ready
    kubernetes cluster.local in-addr.arpa ip6.arpa {
      pods insecure
      fallthrough in-addr.arpa ip6.arpa
      ttl 30
      endpoint https://api-server:6443
      tls "" "" /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
    }
    prometheus :9153
    forward . 8.8.8.8 {
      max_concurrent 1000
    }
    cache 30
    loop
    reload
    loadbalance
}
Plugin Purpose
kubernetes Resolves *.cluster.local by querying the API server over TLS
forward Forwards non-cluster queries to 8.8.8.8 (Google DNS)
prometheus Exposes metrics on port 9153
cache Caches responses for 30 seconds
health / ready Health and readiness endpoints for probes
loop Detects and breaks forwarding loops

Certificate Generation

The scripts/generate-certs.sh script generates all TLS material before the cluster starts. Certificates are persisted in .rusternetes/certs/ and reused across restarts.

ServiceAccount Signing Keys

An RSA 2048-bit key pair is generated for signing and verifying ServiceAccount JWT tokens (RS256):

File Purpose
.rusternetes/certs/sa.key RSA private key — used by the API server to sign JWTs
.rusternetes/certs/sa.pub RSA public key — used to verify SA tokens

API Server TLS Certificate

A self-signed EC P-256 certificate valid for 10 years. The certificate includes Subject Alternative Names (SANs) for all addresses the API server may be reached at: 

        

          
          
          
          certificate SANs
        

        # DNS SANs
DNS.1 = localhost
DNS.2 = api-server
DNS.3 = rusternetes-api-server
DNS.4 = kubernetes
DNS.5 = kubernetes.default
DNS.6 = kubernetes.default.svc
DNS.7 = kubernetes.default.svc.cluster.local

# IP SANs
IP.1 = 127.0.0.1
IP.2 = 10.96.0.1          # kubernetes ClusterIP
IP.3 = 172.18.0.2         # Docker bridge IPs
IP.4 = 172.18.0.3
IP.5 = 172.18.0.4
IP.6 = 172.18.0.5
DOCKER BRIDGE IPS

The Docker bridge IPs (172.18.0.2–5) must be included in the certificate SANs. Without them, kubelet and other components connecting to the API server via the Docker network will fail TLS verification. If your Docker network uses different IPs, regenerate the certificates.

Property Value
Algorithm EC P-256 (prime256v1)
Validity 10 years (3650 days)
Subject CN=rusternetes-api, O=Rusternetes, C=US
Key Usage digitalSignature, keyEncipherment, dataEncipherment, keyCertSign
Extended Key Usage serverAuth, clientAuth
CA TRUE (self-signed root)

Generated Files

        

          
          
          
          file layout
        

        .rusternetes/certs/
  api-server.crt       # API server TLS certificate
  api-server.key       # API server TLS private key
  ca.crt               # CA certificate (copy of api-server.crt)
  sa.key               # ServiceAccount signing key (RSA 2048)
  sa.pub               # ServiceAccount verification key

KUBECONFIG Generation

The bootstrap process generates a kubeconfig file at ~/.kube/rusternetes-config that points to the local API server: 

        

          
          
          
          terminal
        

        # Set KUBECONFIG to use the rusternetes cluster
export KUBECONFIG=~/.kube/rusternetes-config

# Verify connectivity
kubectl --insecure-skip-tls-verify get nodes

Manual Bootstrap Steps

If you prefer to bootstrap manually or need to customize the process: 

        

          
          
          
          terminal
        

        # 1. Generate certificates (if not already present)
bash scripts/generate-certs.sh

# 2. Start the cluster
docker compose up -d

# 3. Wait for the API server to be ready
until kubectl --insecure-skip-tls-verify get --raw /healthz 2>/dev/null; do
  sleep 2
done

# 4. Generate ServiceAccount tokens
bash scripts/generate-default-serviceaccounts.sh

# 5. Apply ServiceAccounts
kubectl --insecure-skip-tls-verify apply -f .rusternetes/default-serviceaccounts.yaml

# 6. Apply bootstrap resources
kubectl --insecure-skip-tls-verify apply -f bootstrap-cluster.yaml

# 7. Verify
kubectl --insecure-skip-tls-verify get pods -A
kubectl --insecure-skip-tls-verify get services -A
REGENERATING CERTIFICATES

To regenerate certificates, delete the existing files and re-run the script: rm .rusternetes/certs/api-server.crt .rusternetes/certs/api-server.key && bash scripts/generate-certs.sh

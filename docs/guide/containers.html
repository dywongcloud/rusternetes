Container Runtime

The Rusternetes kubelet manages containers via the Docker API (through the bollard Rust library). This page covers the container lifecycle, pause containers, probes, lifecycle hooks, init and sidecar containers, resource limits, image pulling, logging, exec, attach, and port forwarding.

Container Lifecycle

Each container in a pod goes through a well-defined lifecycle managed by the kubelet: 

        

          
          
          
          container lifecycle
        

        create → start → [postStart hook] → running → [preStop hook] → SIGTERM → [grace period] → SIGKILL → terminated

Container States (reported in pod status):
  Waiting     — Container is not yet running (pulling image, etc.)
  Running     — Container is executing
  Terminated  — Container has stopped (exit code, reason, signal)

Container Restarts

The kubelet restarts individual containers, not entire pods. Restart behavior depends on the pod's restartPolicy:

restartPolicyBehavior
AlwaysRestart the container regardless of exit code. Used for Deployments.
OnFailureRestart only if the container exits with a non-zero exit code. Used for Jobs.
NeverNever restart. Let the pod transition to Failed or Succeeded.
Individual container restarts

The kubelet tracks restartCount per container across sync cycles. When a container needs restarting, only that container is recreated — other containers in the pod continue running. The pod's overall status reflects the aggregate state of all containers.

Pause Container

Every pod starts with a "pause" container (also called the sandbox or infra container). This container:

  • Creates the pod's network namespace.
  • Holds the namespace open even if all app containers restart.
  • All other containers in the pod join the pause container's network namespace, sharing the same IP address and port space.
  • Is created first and must reach running state before app containers start.
CRI sandbox semantics

The start_pause_container function blocks until the container reaches a running state. This ensures the network namespace is fully established before application containers attempt to join it.

Probes

Probes periodically check the health of containers. Three types of probes serve different purposes:

Probe Types

ProbePurposeOn Failure
livenessProbe Detect deadlocked containers Container is killed and restarted (subject to restartPolicy)
readinessProbe Detect containers not ready to serve Container is removed from Service endpoints (traffic stops flowing)
startupProbe Detect slow-starting containers Container is killed. While startup probe is active, liveness and readiness probes are disabled.

Probe Handlers

HandlerDescriptionSuccess Criteria
httpGet HTTP GET request to the container Status code 200–399
tcpSocket TCP connection to the container port Connection established
exec Run a command inside the container Exit code 0
grpc gRPC health check (grpc.health.v1.Health) SERVING response

Probe Configuration

FieldDefaultDescription
initialDelaySeconds0Seconds to wait after container start before first probe.
periodSeconds10How often to perform the probe.
timeoutSeconds1Timeout for a single probe attempt.
successThreshold1Consecutive successes required to mark as healthy.
failureThreshold3Consecutive failures required to mark as unhealthy.
 
        

          
          
          
          probes.yaml
        

        containers:
  - name: app
    image: myapp:v1
    ports:
      - containerPort: 8080
    startupProbe:
      httpGet:
        path: /healthz
        port: 8080
      failureThreshold: 30
      periodSeconds: 10
      # Allow up to 5 min for startup (30 * 10s)
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
        scheme: HTTP
        httpHeaders:
          - name: X-Custom-Header
            value: Probe
      initialDelaySeconds: 0
      periodSeconds: 15
      timeoutSeconds: 3
      failureThreshold: 3
    readinessProbe:
      tcpSocket:
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      successThreshold: 1
      
        

          
          
          
          exec-probe.yaml
        

        livenessProbe:
  exec:
    command:
      - /bin/sh
      - -c
      - "pg_isready -U postgres"
  periodSeconds: 30
  failureThreshold: 3
      
        

          
          
          
          grpc-probe.yaml
        

        readinessProbe:
  grpc:
    port: 50051
    service: my.custom.Service   # Optional, defaults to ""
  periodSeconds: 10

Lifecycle Hooks

Containers can define hooks that run at specific points in their lifecycle.

Hook Types

HookWhenBehavior
postStart Immediately after container creation Runs concurrently with the container's entrypoint. Blocks the container from becoming Ready until it completes. If it fails, the container is killed.
preStop Before SIGTERM is sent Runs before the container is terminated. The grace period starts when preStop begins. After preStop completes (or times out), SIGTERM is sent.
postStart failure kills the container

If a postStart hook fails, Kubernetes kills the container. The container will be restarted according to the pod's restartPolicy. This is often overlooked — do not use postStart for non-essential tasks.

Hook Handlers

HandlerDescription
execRun a command inside the container.
httpGetSend an HTTP GET request to the container.
tcpSocketOpen a TCP connection to the container port.
sleepWait for a specified duration (seconds).
 
        

          
          
          
          hooks.yaml
        

        containers:
  - name: app
    image: myapp:v1
    lifecycle:
      postStart:
        exec:
          command: ["/bin/sh", "-c", "echo 'App started' >> /var/log/app.log"]
      preStop:
        httpGet:
          path: /shutdown
          port: 8080
    # OR use sleep for a simple delay:
    # preStop:
    #   sleep:
    #     seconds: 10
    stopSignal: SIGTERM         # Custom stop signal (default SIGTERM)

Init Containers

Init containers run before app containers start. They run sequentially — each must complete successfully (exit code 0) before the next begins. If an init container fails, the pod is restarted according to its restartPolicy.

Use Cases

  • Wait for a database or service to become available.
  • Download configuration or data from an external source.
  • Set file permissions or generate configs before the main app starts.
  • Run database migrations.
        

          
          
          
          init-containers.yaml
        

        spec:
  initContainers:
    - name: wait-for-db
      image: busybox:1.36
      command: ['sh', '-c', 'until nc -z db-svc 5432; do sleep 2; done']
    - name: migrate
      image: myapp:migrate
      command: ['./migrate', '--up']
  containers:
    - name: app
      image: myapp:v1

Sidecar Containers

Sidecar containers are init containers with restartPolicy: Always. They start before the main containers (as part of the init sequence) but continue running alongside them for the pod's lifetime. 

        

          
          
          
          sidecar.yaml
        

        spec:
  initContainers:
    - name: log-shipper
      image: fluentd:v1.16
      restartPolicy: Always     # Makes this a sidecar
      volumeMounts:
        - name: logs
          mountPath: /var/log/app
  containers:
    - name: app
      image: myapp:v1
      volumeMounts:
        - name: logs
          mountPath: /var/log/app
  volumes:
    - name: logs
      emptyDir: {}

Resource Limits

The kubelet passes CPU and memory limits to Docker when creating containers. Docker enforces these via Linux cgroups.

ResourceDocker ParameterEnforcement
CPU requestsCpuSharesProportional CPU time in contention.
CPU limitsNanoCpusHard CPU cap. Container is throttled.
Memory limitsMemoryHard memory cap. Container is OOMKilled if exceeded.

Image Pulling

The kubelet pulls container images through the Docker daemon. The imagePullPolicy field controls when images are pulled.

PolicyBehavior
AlwaysPull the image every time the container starts. Default for :latest tag.
IfNotPresentPull only if the image is not already present locally. Default for tagged images.
NeverNever pull. Container fails if image is not present locally.

Container Logs

Container stdout and stderr are captured by Docker and accessible via the kubelet's logs endpoint. 

        

          
          
          
          terminal
        

        # View logs from a pod
kubectl logs my-pod

# Follow logs in real time
kubectl logs my-pod --follow

# Last N lines
kubectl logs my-pod --tail=100

# Logs since a duration
kubectl logs my-pod --since=1h

# Logs with timestamps
kubectl logs my-pod --timestamps

# Logs from a specific container in a multi-container pod
kubectl logs my-pod --container=sidecar

# Logs from all containers
kubectl logs my-pod --all-containers

# Logs from the previous container instance (after restart)
kubectl logs my-pod --previous

Exec

Execute a command inside a running container. Uses WebSocket-based streaming for stdin/stdout/stderr with optional TTY support. 

        

          
          
          
          terminal
        

        # Run a command
kubectl exec my-pod -- ls /app

# Interactive shell
kubectl exec -it my-pod -- /bin/sh

# Exec in a specific container
kubectl exec -it my-pod --container=app -- /bin/bash

# Run a command and capture output
kubectl exec my-pod -- cat /etc/hostname

Attach

Attach to a running container's stdin/stdout. Unlike exec, attach connects to the container's main process rather than starting a new one. 

        

          
          
          
          terminal
        

        # Attach to a container (stdout only)
kubectl attach my-pod

# Attach with stdin and TTY
kubectl attach -it my-pod

# Attach to a specific container
kubectl attach -it my-pod --container=app

Port Forwarding

Forward local ports to a pod's ports. Useful for accessing services that are not exposed externally. 

        

          
          
          
          terminal
        

        # Forward local port 8080 to pod port 80
kubectl port-forward my-pod 8080:80

# Forward multiple ports
kubectl port-forward my-pod 8080:80 9090:9090

# Forward to a service instead of a pod
kubectl port-forward svc/web 8080:80

# Listen on all interfaces
kubectl port-forward --address=0.0.0.0 my-pod 8080:80

Debugging Containers

kubectl debug creates ephemeral debug containers in a running pod or copies a pod with modified settings for troubleshooting. 

        

          
          
          
          terminal
        

        # Add an ephemeral debug container to a running pod
kubectl debug my-pod -it --image=busybox

# Share the process namespace with a target container
kubectl debug my-pod -it --image=busybox --target=app

# Copy a pod and change the command (for distroless images)
kubectl debug my-pod -it --copy-to=my-pod-debug --image=ubuntu -- /bin/bash

# Debug a node
kubectl debug node/node-1 -it --image=ubuntu

hostPort

Containers can bind directly to a port on the host node. The scheduler's HostPort admission plugin rejects pods with conflicting host port allocations (setting the pod's phase to Failed). 

        

          
          
          
          hostport.yaml
        

        containers:
  - name: nginx
    image: nginx:1.27
    ports:
      - containerPort: 80
        hostPort: 80           # Bind to node port 80
        protocol: TCP
hostPort limitations

Only one pod per node can use a given hostPort. If two pods request the same hostPort on the same node, the second pod is rejected with phase Failed. Prefer Services with NodePort for most use cases.

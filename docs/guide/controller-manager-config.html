Controller Manager Configuration

The controller manager runs 31 reconciliation controllers as concurrent tokio tasks. Each controller watches resources in storage and drives them toward their desired state.

CLI Flags

Flag Type Default Description
--etcd-servers String http://localhost:2379 Comma-separated list of etcd endpoints
--storage-backend String etcd Storage backend: etcd or sqlite
--data-dir String ./data/rusternetes.db SQLite database path (only with --storage-backend=sqlite)
--log-level String info Log level: trace, debug, info, warn, error
--sync-interval u64 5 Controller sync interval in seconds
--cloud-provider String auto-detected Cloud provider: aws, gcp, azure, or none
--cluster-name String rusternetes Cluster name for cloud provider resources
--cloud-region String — Cloud provider region
--enable-leader-election bool false Enable leader election for HA
--leader-election-identity String auto-generated UUID Unique identity for this instance
--leader-election-lock-key String /rusternetes/controller-manager/leader etcd key used for the leader election lock
--leader-election-lease-duration u64 15 Leader election lease duration in seconds

Usage Examples

        

          
          
          
          terminal
        

        # Standard deployment with etcd
rusternetes-controller-manager \
  --etcd-servers http://etcd:2379 \
  --sync-interval 5

# With cloud provider
rusternetes-controller-manager \
  --etcd-servers http://etcd:2379 \
  --cloud-provider aws \
  --cloud-region us-east-1

# HA with leader election
rusternetes-controller-manager \
  --etcd-servers http://etcd:2379 \
  --enable-leader-election \
  --leader-election-identity cm-1

All 31 Controllers

Each controller runs as an independent tokio task. Some use watch-based reconciliation for immediate response to changes; others use periodic polling.

# Controller Reconciles
1 Deployment Creates/updates ReplicaSets, manages rollouts and rollbacks
2 ReplicaSet Ensures the correct number of pod replicas exist
3 ReplicationController Legacy replication controller (v1 API)
4 StatefulSet Ordered pod creation/deletion with stable network identities
5 DaemonSet Ensures a pod runs on every (or selected) node
6 Job Runs pods to completion, tracks completions and failures
7 CronJob Creates Jobs on a cron schedule
8 Endpoints Populates Endpoints objects from Service selectors and pod IPs (watch-based)
9 EndpointSlice Populates EndpointSlice objects for services (watch-based)
10 Service Manages service lifecycle and status updates (watch-based)
11 ServiceAccount Creates default ServiceAccount in new namespaces (watch-based)
12 Namespace Handles namespace deletion, cleans up contained resources (watch-based)
13 Node Manages node lifecycle, heartbeats, and conditions (watch-based)
14 PV Binder Binds PersistentVolumeClaims to PersistentVolumes
15 Dynamic Provisioner Dynamically provisions PersistentVolumes for unbound PVCs
16 Volume Snapshot Manages VolumeSnapshot lifecycle and binding
17 Volume Expansion Handles PVC resize operations
18 ResourceQuota Tracks resource usage against quota limits (cpu, memory, ephemeral-storage)
19 ResourceClaim Manages dynamic resource allocation claims
20 HPA Horizontal Pod Autoscaler — scales based on CPU/memory/custom metrics
21 VPA Vertical Pod Autoscaler — recommends resource requests
22 PodDisruptionBudget Tracks disruption budget status for voluntary disruptions
23 LoadBalancer Provisions external load balancers via cloud provider
24 Ingress Manages Ingress resources and updates status
25 NetworkPolicy Manages network policy lifecycle and status
26 CRD Manages CustomResourceDefinition lifecycle, sets status conditions
27 CertificateSigningRequest Auto-approves and signs CSRs
28 Garbage Collector Deletes orphaned resources based on ownerReferences
29 TTL Deletes finished Jobs/Pods after ttlSecondsAfterFinished
30 Taint Eviction Evicts pods from nodes with NoExecute taints (watch-based)
31 Events Manages event lifecycle and garbage collection
WATCH-BASED CONTROLLERS

Controllers marked as "watch-based" use storage watch streams for immediate reactivity instead of periodic polling. They respond to changes within milliseconds rather than waiting for the next sync interval.

Leader Election

When leader election is enabled, all 31 controllers wait in follower mode until the instance acquires the etcd lease. If leadership is lost, controllers stop and wait to re-acquire. This prevents split-brain scenarios in HA deployments. 

        

          
          
          
          leader election flow
        

        # Instance starts in follower mode
Follower  ---[acquire lease]--->  Leader  ---[run controllers]--->  Active

# If lease is lost
Active    ---[lose lease]--->  Follower  ---[wait]--->  ...

# Lease timing
Lease duration:    15s  (--leader-election-lease-duration)
Renew interval:    5s   (lease_duration / 3)
Retry interval:    2s   (fixed)

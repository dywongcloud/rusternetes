Cloud Providers

Rūsternetes integrates with AWS, GCP, and Azure for cloud-native load balancing, node management, and infrastructure automation. AWS is fully implemented; GCP and Azure are planned.

Provider Detection

The cloud provider can be configured explicitly via the --cloud-provider flag on the controller-manager, or auto-detected from environment variables. 

        

          
          
          
          terminal
        

        # Explicit provider selection
./rusternetes-controller-manager --cloud-provider aws
./rusternetes-controller-manager --cloud-provider gcp
./rusternetes-controller-manager --cloud-provider azure
./rusternetes-controller-manager --cloud-provider none

Auto-detection

When no --cloud-provider flag is set, the controller-manager checks for provider-specific environment variables:

Provider Environment Variable Detected When
AWS AWS_REGION or AWS_DEFAULT_REGION Either variable is set
GCP GCP_PROJECT or GOOGLE_CLOUD_PROJECT Either variable is set
Azure AZURE_SUBSCRIPTION_ID Variable is set
Any CLOUD_PROVIDER Explicit override: aws, gcp, azure, or none

Feature flags

Cloud provider support is gated behind Cargo feature flags. Only the providers you need are compiled into the binary: 

        

          
          
          
          terminal
        

        # Build with AWS support only
cargo build --features aws

# Build with all cloud providers
cargo build --features aws,gcp,azure

# Build without any cloud provider (default)
cargo build

AWS (Implemented)

The AWS cloud provider creates and manages Elastic Load Balancers for Kubernetes Services of type LoadBalancer. It uses the AWS SDK for Rust to interact with the EC2 and ELBv2 APIs.

Configuration

Environment Variable Required Description
AWS_REGION Yes AWS region (e.g., us-east-1)
AWS_VPC_ID No VPC ID. Auto-detected from EC2 instance metadata if not set
AWS_SUBNET_IDS No Comma-separated subnet IDs. Auto-detected if not set
AWS_ACCESS_KEY_ID Yes AWS access key (or use IAM role / instance profile)
AWS_SECRET_ACCESS_KEY Yes AWS secret key (or use IAM role / instance profile)

Network Load Balancer (NLB)

When a Service of type LoadBalancer is created, the AWS provider:

  1. Creates a Network Load Balancer (NLB) in the configured VPC and subnets
  2. Creates a target group for each service port with IPv4 address type
  3. Registers cluster nodes by their IP addresses as targets
  4. Creates listeners that forward traffic from the NLB to the target groups
  5. Updates the Service's status.loadBalancer.ingress with the NLB hostname

Naming and tagging

Property Value
NLB name {cluster}-{namespace}-{service} (max 32 characters)
Target group name {cluster}-{namespace}-{service}-{port}
Tag: kubernetes.io/cluster Cluster name
Tag: managed-by rusternetes

Annotations

The following Service annotations are supported for AWS load balancers:

Annotation Values Description
service.beta.kubernetes.io/aws-load-balancer-internal "true" / "false" Create an internal NLB (not internet-facing). Default: false 
        

          
          
          
          service-lb.yaml
        

        apiVersion: v1
kind: Service
metadata:
  name: my-app
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
spec:
  type: LoadBalancer
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
VPC auto-detection

When running on EC2, the provider auto-detects the VPC and subnets via the EC2 instance metadata service (IMDS). Set AWS_VPC_ID and AWS_SUBNET_IDS explicitly when running outside of EC2 or when you need to target specific subnets.

GCP (Planned)

GCP cloud provider support is currently a stub implementation. The configuration structure is defined but load balancer creation is not yet implemented.

Configuration

Environment Variable Description
GCP_PROJECT or GOOGLE_CLOUD_PROJECT GCP project ID
GCP_REGION GCP region (e.g., us-central1)
Not yet implemented

GCP provider is a stub. Services of type LoadBalancer will be accepted but no GCP resources will be created. Contributions welcome.

Azure (Planned)

Azure cloud provider support is currently a stub implementation. The configuration structure is defined but load balancer creation is not yet implemented.

Configuration

Environment Variable Description
AZURE_SUBSCRIPTION_ID Azure subscription ID
AZURE_RESOURCE_GROUP Azure resource group name
AZURE_LOCATION Azure region (e.g., eastus)
Not yet implemented

Azure provider is a stub. Services of type LoadBalancer will be accepted but no Azure resources will be created. Contributions welcome.

Disabling cloud providers

To run without any cloud provider integration, set the provider to none: 

        

          
          
          
          terminal
        

        ./rusternetes-controller-manager --cloud-provider none

# Or via environment variable
export CLOUD_PROVIDER=none

When no cloud provider is configured, Services of type LoadBalancer will remain in Pending state with no external IP assigned. This is the default behavior for local development clusters.

Quick Start

Get a fully functional Rūsternetes cluster running in 5 minutes using Docker Compose. No Rust toolchain required.

Prerequisites

Platform Requirement
macOS Docker Desktop (recommended) or Podman Desktop
Linux Docker Engine 24+ or rootful Podman 4+

You also need kubectl installed on your host machine. Any recent version works — Rūsternetes is compatible with the standard Kubernetes CLI.

Rust is optional

The Docker Compose setup builds everything inside containers. You only need Rust installed if you want to build from source outside of Docker.

Step-by-Step Setup

1. Clone the repository

        

          
          
          
          terminal
        

        git clone https://github.com/calfonso/rusternetes.git
cd rusternetes

2. Set the volumes path

The kubelet needs a shared directory to mount ConfigMaps, Secrets, and other volumes into pod containers. This path must be absolute and accessible to both Docker and the kubelet container. 

        

          
          
          
          terminal
        

        export KUBELET_VOLUMES_PATH=$(pwd)/.rusternetes/volumes
mkdir -p $KUBELET_VOLUMES_PATH
Podman on macOS

If using Podman Machine on macOS, /tmp is not shared between host and VM. Use a path under your home directory (which is automatically mounted). The $(pwd)/.rusternetes/volumes path works correctly.

3. Build the container images

This compiles each Rūsternetes component inside a Docker build stage. The first build takes 10–15 minutes; subsequent builds use the Docker layer cache and are much faster. 

        

          
          
          
          terminal
        

        docker compose build

4. Start the cluster

        

          
          
          
          terminal
        

        docker compose up -d

This starts all seven services. The API server waits for etcd to be healthy before accepting connections.

5. Bootstrap the cluster

The bootstrap script generates TLS certificates, creates system namespaces (kube-system, kube-public, default), deploys CoreDNS, creates ServiceAccounts, and sets up the kubernetes service. 

        

          
          
          
          terminal
        

        bash scripts/bootstrap-cluster.sh

6. Configure kubectl

        

          
          
          
          terminal
        

        export KUBECONFIG=~/.kube/rusternetes-config

The bootstrap script writes this kubeconfig file automatically. It points to https://localhost:6443 with the generated TLS credentials.

7. Verify the cluster

        

          
          
          
          terminal
        

        kubectl get nodes

You should see two nodes in Ready status:

        

          
          
          
          output
        

        NAME     STATUS   ROLES    AGE   VERSION
node-1   Ready    <none>   30s   v1.35.0-rusternetes
node-2   Ready    <none>   30s   v1.35.0-rusternetes

Try deploying a workload:

        

          
          
          
          terminal
        

        kubectl create deployment nginx --image=nginx
kubectl get pods -w

What You Get

The Docker Compose cluster runs seven services that mirror a real Kubernetes cluster:

Service Container Role
etcd rusternetes-etcd Distributed key-value store for all cluster state
api-server rusternetes-api-server HTTPS REST API on port 6443 (TLS)
scheduler rusternetes-scheduler Assigns pods to nodes using filter/score plugins
controller-manager rusternetes-controller-manager Runs 31 reconciliation controllers
kubelet rusternetes-kubelet Node agent on node-1, manages containers via Docker
kubelet2 rusternetes-kubelet2 Node agent on node-2, same as above
kube-proxy rusternetes-kube-proxy iptables-based service routing (host network mode)

After bootstrapping, the cluster also runs CoreDNS as a pod in the kube-system namespace, providing in-cluster DNS resolution at 10.96.0.10.

Common Operations

View logs

        

          
          
          
          terminal
        

        # All services
docker compose logs -f

# Single service
docker compose logs -f api-server
docker compose logs -f kubelet
docker compose logs -f scheduler

Rebuild after code changes

        

          
          
          
          terminal
        

        # Rebuild and restart a single service
docker compose build api-server
docker compose up -d api-server

# Rebuild everything
docker compose build
docker compose up -d

Stop the cluster

        

          
          
          
          terminal
        

        # Stop containers (preserves data)
docker compose down

# Stop and delete all data (clean restart)
docker compose down -v

Clean restart

If you need to wipe the cluster completely and start fresh: 

        

          
          
          
          terminal
        

        # Remove containers, volumes, and orphan pod containers
docker compose down -v
docker rm -f $(docker ps -aq --filter "label=rusternetes") 2>/dev/null || true

# Start fresh
docker compose up -d
bash scripts/bootstrap-cluster.sh

Troubleshooting

Port 6443 already in use

Another process (often a real Kubernetes or minikube) is listening on port 6443. Stop it or change the port mapping in docker-compose.yml: 

        

          
          
          
          docker-compose.yml
        

        # Change the host port (left side) to any free port
ports:
  - "7443:6443"

Then update your kubeconfig to use the new port:

        

          
          
          
          terminal
        

        kubectl config set-cluster rusternetes --server=https://localhost:7443

Cannot connect to API server

If kubectl get nodes returns a connection error:

  • Check that the API server is running: docker compose ps
  • Check the API server logs: docker compose logs api-server
  • Verify etcd is healthy: docker compose logs etcd
  • Make sure KUBECONFIG is set: echo $KUBECONFIG
  • Confirm the kubeconfig file exists: cat ~/.kube/rusternetes-config

First build is slow

The initial docker compose build compiles the entire Rust workspace from scratch, which takes 10–15 minutes depending on your hardware. Subsequent builds use the Docker layer cache and only recompile changed crates. To speed things up:

  • Allocate more CPUs and memory to Docker Desktop (Settings → Resources)
  • Rebuild only the service you changed: docker compose build api-server
  • Consider building from source natively for faster iteration (see Installation)

Podman: permission denied

On macOS with Podman, you may see permission errors for volume mounts. Ensure:

  • Podman Machine is running in rootful mode: podman machine set --rootful
  • The volumes path is under your home directory, not /tmp
  • The Podman socket is accessible: podman info

Pods stuck in Pending

If pods stay in Pending after deployment:

  • Check the scheduler is running: docker compose ps scheduler
  • Check scheduler logs: docker compose logs scheduler
  • Verify nodes are Ready: kubectl get nodes
  • Describe the pod for events: kubectl describe pod <name>

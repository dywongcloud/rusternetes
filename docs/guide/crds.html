Custom Resource Definitions

CRDs extend the Kubernetes API with your own resource types. Rusternetes supports the full CRD lifecycle: creation, schema validation, CRUD operations, watch streams, status and scale subresources, schema defaulting, structural pruning, and strict field validation.

Creating a CRD

CRD Structure

FieldDescription
groupAPI group (e.g., stable.example.com). Used in API paths: /apis/{group}/{version}/{plural}.
names.kindPascalCase kind (e.g., CronTab).
names.pluralLowercase plural (e.g., crontabs). Used in API paths.
names.singularLowercase singular (e.g., crontab). Used by kubectl.
names.shortNamesShort aliases (e.g., ["ct"]). Used by kubectl.
names.categoriesGroups like ["all"] so kubectl get all includes this resource.
scopeNamespaced or Cluster.
versionsArray of version definitions. Each has name, served, storage, and schema.

Full CRD Example

        

          
          
          
          crontab-crd.yaml
        

        apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: crontabs.stable.example.com
spec:
  group: stable.example.com
  names:
    kind: CronTab
    plural: crontabs
    singular: crontab
    shortNames:
      - ct
    categories:
      - all
  scope: Namespaced
  versions:
    - name: v1
      served: true
      storage: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              required: ["cronSpec", "image"]
              properties:
                cronSpec:
                  type: string
                  pattern: "^(\\d+|\\*) (\\d+|\\*) (\\d+|\\*) (\\d+|\\*) (\\d+|\\*)$"
                image:
                  type: string
                replicas:
                  type: integer
                  minimum: 1
                  maximum: 10
                  default: 1
            status:
              type: object
              properties:
                replicas:
                  type: integer
                labelSelector:
                  type: string
      subresources:
        status: {}
        scale:
          specReplicasPath: .spec.replicas
          statusReplicasPath: .status.replicas
          labelSelectorPath: .status.labelSelector
      additionalPrinterColumns:
        - name: Schedule
          type: string
          jsonPath: .spec.cronSpec
        - name: Image
          type: string
          jsonPath: .spec.image
        - name: Replicas
          type: integer
          jsonPath: .spec.replicas
        - name: Age
          type: date
          jsonPath: .metadata.creationTimestamp

OpenAPI v3 Schema

The openAPIV3Schema validates custom resource instances. Schema validation runs against schema.properties["spec"], not the top-level schema directly.

Supported Validation Keywords

KeywordDescription
typeJSON type: object, array, string, integer, number, boolean.
formatString format: date-time, date, int32, int64, float, double, byte.
patternRegex pattern for strings.
requiredArray of required property names.
propertiesMap of property name to sub-schema.
additionalPropertiesSchema or boolean for properties not in properties.
minimum / maximumNumeric bounds (inclusive).
exclusiveMinimum / exclusiveMaximumNumeric bounds (exclusive).
minLength / maxLengthString length bounds.
enumArray of allowed values.
oneOf / anyOf / allOf / notSchema composition.
itemsSchema for array elements.
minItems / maxItemsArray length bounds.
defaultDefault value applied on read.
nullableAllow null values.
descriptionHuman-readable description.
Schema validation scope

Validation is applied to spec and status sub-schemas within openAPIV3Schema. The top-level schema wraps spec, status, and metadata — validate against the appropriate sub-property, not the root.

CRUD Operations

Custom Resource Instance

        

          
          
          
          my-crontab.yaml
        

        apiVersion: stable.example.com/v1
kind: CronTab
metadata:
  name: my-cron
  namespace: default
spec:
  cronSpec: "*/5 * * * *"
  image: my-cron-image:v1
  replicas: 3
      
        

          
          
          
          terminal
        

        # Create the CRD
kubectl apply -f crontab-crd.yaml

# Create a custom resource
kubectl apply -f my-crontab.yaml

# Get custom resources (using plural, singular, or short name)
kubectl get crontabs
kubectl get crontab my-cron
kubectl get ct

# Get with custom columns from additionalPrinterColumns
kubectl get ct -o wide

# Describe
kubectl describe ct my-cron

# Update
kubectl apply -f my-crontab.yaml

# Patch (all three types supported)
kubectl patch ct my-cron --type=merge -p '{"spec":{"replicas":5}}'
kubectl patch ct my-cron --type=json -p '[{"op":"replace","path":"/spec/replicas","value":5}]'
kubectl apply --server-side -f my-crontab.yaml  # Server-side apply

# Delete
kubectl delete ct my-cron

Watch

Custom resources support real-time watch streams, identical to built-in resources. 

        

          
          
          
          terminal
        

        # Watch for changes
kubectl get ct -w

# Watch with raw API (for debugging)
curl -k "https://localhost:6443/apis/stable.example.com/v1/crontabs?watch=true&timeoutSeconds=60"

Watch Event Types

TypeDescription
ADDEDResource created or first seen in initial list.
MODIFIEDResource updated.
DELETEDResource deleted.
BOOKMARKCheckpoint with the latest resourceVersion. Used by clients to resume watches.

Watch Query Parameters

ParameterDescription
watch=trueEnable watch mode.
resourceVersionStart watching from this version. 0 = start from current state.
timeoutSecondsServer-side timeout for the watch connection.
sendInitialEvents=trueSend ADDED events for all existing resources before streaming changes.

Status Subresource

When subresources.status: {} is set in the CRD version, a separate /status endpoint is created. Updates to the main resource ignore changes to .status, and updates to /status ignore changes to .spec. 

        

          
          
          
          terminal
        

        # Get status
kubectl get ct my-cron -o jsonpath='{.status}'

# Update status (typically done by a controller)
curl -k -X PUT \
  "https://localhost:6443/apis/stable.example.com/v1/namespaces/default/crontabs/my-cron/status" \
  -H "Content-Type: application/json" \
  -d '{"apiVersion":"stable.example.com/v1","kind":"CronTab","metadata":{"name":"my-cron","namespace":"default"},"status":{"replicas":3}}'

# Patch status
kubectl patch ct my-cron --subresource=status --type=merge -p '{"status":{"replicas":3}}'

Scale Subresource

When subresources.scale is configured, the CRD gets a /scale endpoint compatible with kubectl scale and HPA.

Scale Configuration

FieldDescription
specReplicasPathJSONPath to the replica count in spec (e.g., .spec.replicas).
statusReplicasPathJSONPath to the replica count in status (e.g., .status.replicas).
labelSelectorPathOptional. JSONPath to the label selector string in status.
Scale selector format

The scale subresource returns the label selector as a plain string (key=value,key2=value2), not JSON. kubectl expects this format and will crash on JSON-encoded selectors. 

        

          
          
          
          terminal
        

        # Scale a custom resource
kubectl scale ct my-cron --replicas=5

# Get scale
kubectl get ct my-cron --subresource=scale

Schema Defaulting

Fields with a default value in the schema have that value applied when the resource is read if the field is not present. In the CronTab example above, spec.replicas defaults to 1.

Structural Pruning

Unknown fields — fields not declared in the schema — are automatically removed when the resource is created or updated. This ensures custom resources conform strictly to the declared schema.

To preserve unknown fields (e.g., for free-form JSON), add the extension: 

        

          
          
          
          preserve-unknown.yaml
        

        properties:
  spec:
    type: object
    x-kubernetes-preserve-unknown-fields: true
    properties:
      knownField:
        type: string
      # Additional fields are preserved, not pruned

Strict Field Validation

Clients can request strict validation to reject requests with unknown or duplicate fields: 

        

          
          
          
          terminal
        

        # Strict validation via query parameter
curl -k -X POST \
  "https://localhost:6443/apis/stable.example.com/v1/namespaces/default/crontabs?fieldValidation=Strict" \
  -H "Content-Type: application/json" \
  -d @my-crontab.json

# kubectl uses Strict by default with --validate=strict
kubectl apply -f my-crontab.yaml --validate=strict
fieldValidationBehavior
StrictReject unknown fields. Return error.
WarnAccept but return warnings for unknown fields.
IgnoreSilently drop unknown fields (default behavior).

Storage

Custom resources are stored in the backend (etcd or SQLite) at the key path: 

        

          
          
          
          storage key format
        

        /registry/{group_underscored}_{plural}/{namespace}/{name}

# Example:
/registry/stable_example_com_crontabs/default/my-cron
Raw JSON storage

Custom resources are stored as raw JSON, not as typed Rust structs. This preserves nested schemas, enum values, and extension fields that would be lost in a serialize/deserialize round-trip through typed structures.

Schema Serialization

The CRD's JSONSchemaProps uses skip_serializing_if helpers to omit empty strings and false booleans. This is required because the Kubernetes API distinguishes between "field absent" and "field set to zero value" for schema properties.

RustJSON
nullable: Some(false)Omitted (not "nullable": false)
nullable: Some(true)"nullable": true
format: Some("")Omitted (not "format": "")
format: Some("date-time")"format": "date-time"

kubectl explain

CRDs support kubectl explain for documentation. The API server adds the x-kubernetes-group-version-kind extension to the schema so kubectl can discover the resource type. 

        

          
          
          
          terminal
        

        # Explain custom resource fields
kubectl explain crontab
kubectl explain crontab.spec
kubectl explain crontab.spec.cronSpec
kubectl explain crontab --recursive

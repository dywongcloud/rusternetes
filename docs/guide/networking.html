Networking

Rusternetes implements the Kubernetes networking model: every pod gets its own IP address, Services provide stable endpoints for sets of pods, and kube-proxy programs iptables rules for in-cluster routing. CoreDNS provides service discovery via DNS.

Services

A Service provides a stable virtual IP (ClusterIP) and DNS name for a set of pods selected by labels. Kubernetes supports five Service types, each building on the previous.

Service Types

TypeDescriptionAccessible From
ClusterIP Default. Allocates a virtual IP from the service CIDR (10.96.0.0/12). Only reachable within the cluster. Inside cluster
NodePort Extends ClusterIP. Opens a port (30000–32767) on every node that forwards to the service. External via <NodeIP>:<NodePort>
LoadBalancer Extends NodePort. Provisions an external load balancer (cloud provider dependent). External via load balancer IP
ExternalName Maps the service to a DNS CNAME record. No proxy, no ClusterIP. DNS alias only
Headless Set clusterIP: None. Returns individual pod IPs in DNS A records. Used with StatefulSets. DNS direct to pods

Key Spec Fields

FieldTypeDescription
typestringService type. Default ClusterIP.
selectormapLabel selector for target pods. Omit for ExternalName or manual Endpoints.
portsarrayPort mappings: port (service port), targetPort (pod port), protocol (TCP/UDP), nodePort (for NodePort/LoadBalancer).
clusterIPstringVirtual IP. Set to None for headless. Leave empty for auto-allocation.
externalTrafficPolicystringCluster (default, SNAT to any node) or Local (preserve source IP, only route to local pods).
internalTrafficPolicystringCluster (default) or Local (only route to pods on the same node).
sessionAffinitystringNone (default) or ClientIP. Sticky sessions by source IP.
sessionAffinityConfigobjectclientIP.timeoutSeconds — duration of sticky session (default 10800 = 3 hours).
externalNamestringDNS name for ExternalName services.
publishNotReadyAddressesbooleanInclude unready pods in Endpoints. Used for StatefulSet headless services.
ipFamiliesarray["IPv4"], ["IPv6"], or both for dual-stack.
ipFamilyPolicystringSingleStack, PreferDualStack, or RequireDualStack.

Examples

        

          
          
          
          clusterip-service.yaml
        

        apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: 8080
      protocol: TCP
      
        

          
          
          
          nodeport-service.yaml
        

        apiVersion: v1
kind: Service
metadata:
  name: web-external
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080    # 30000-32767, or omit for auto-assign
      
        

          
          
          
          headless-service.yaml
        

        apiVersion: v1
kind: Service
metadata:
  name: db-headless
spec:
  clusterIP: None
  selector:
    app: db
  ports:
    - port: 5432
  publishNotReadyAddresses: true
      
        

          
          
          
          terminal
        

        # Create services via kubectl
kubectl expose deployment web --port=80 --target-port=8080 --type=ClusterIP
kubectl expose deployment web --port=80 --type=NodePort --name=web-np
kubectl create service externalname ext-db --external-name=db.example.com

# List services
kubectl get services
kubectl get svc -o wide

Endpoints

The Endpoints controller automatically populates an Endpoints object for each Service that has a selector. The Endpoints resource maps the service name to a list of pod IP:port pairs.

Key Behaviors

  • Only pods in Running phase with passing readiness probes are included in the addresses list.
  • Pods that are not ready appear in notReadyAddresses.
  • Terminal pods (Failed or Succeeded phase) are always excluded.
  • Terminating pods (those with a deletionTimestamp) are excluded unless publishNotReadyAddresses is set on the Service.
  • When publishNotReadyAddresses: true, all non-terminal pods appear in addresses regardless of readiness.
        

          
          
          
          terminal
        

        # View endpoints for a service
kubectl get endpoints web

# Detailed view showing addresses and ports
kubectl describe endpoints web

EndpointSlice

EndpointSlice is the scalable successor to Endpoints. Each slice holds up to 100 endpoints, allowing large services (thousands of pods) to update efficiently without rewriting a single massive Endpoints object.

Key Fields

FieldDescription
addressTypeIPv4, IPv6, or FQDN.
endpointsArray of endpoint objects with addresses, conditions (ready, serving, terminating), targetRef, nodeName.
portsArray of port objects. name must always be set (empty string is valid). A nil name causes client crashes.
Port name must be set

EndpointSlice port.name must always be present, even as an empty string. If it is null/nil, kubectl describe will crash with a nil pointer dereference. Rusternetes always sets port names on EndpointSlices. 

        

          
          
          
          terminal
        

        # List EndpointSlices
kubectl get endpointslices

# EndpointSlices for a specific service
kubectl get endpointslices -l kubernetes.io/service-name=web

DNS (CoreDNS)

Rusternetes deploys CoreDNS as a pod in kube-system with a pinned ClusterIP of 10.96.0.10. The kubelet configures each pod's /etc/resolv.conf to point to this address.

DNS Records

Record TypeFormatExample
Service A record {service}.{namespace}.svc.cluster.local web.default.svc.cluster.local → 10.96.0.42
Service SRV record _{port-name}._{protocol}.{service}.{namespace}.svc.cluster.local _http._tcp.web.default.svc.cluster.local
Headless A record {pod-name}.{service}.{namespace}.svc.cluster.local web-0.web-headless.default.svc.cluster.local → pod IP
Pod A record {pod-ip-dashed}.{namespace}.pod.cluster.local 10-244-0-5.default.pod.cluster.local
ExternalName CNAME {service}.{namespace}.svc.cluster.local ext-db.default.svc.cluster.local → db.example.com

DNS Policies

PolicyBehavior
ClusterFirstDefault. Pod DNS queries go to CoreDNS. Falls back to upstream for non-cluster names.
DefaultUse the node's DNS configuration (/etc/resolv.conf from the host).
ClusterFirstWithHostNetLike ClusterFirst, but for pods using hostNetwork: true.
NoneNo auto-configuration. Must provide dnsConfig with nameservers and search domains.

CoreDNS Configuration

        

          
          
          
          Corefile (ConfigMap)
        

        .:53 {
    errors
    health {
        lameduck 5s
    }
    ready
    kubernetes cluster.local in-addr.arpa ip6.arpa {
        pods insecure
        fallthrough in-addr.arpa ip6.arpa
        ttl 30
    }
    forward . /etc/resolv.conf {
        max_concurrent 1000
    }
    cache 30
    loop
    reload
    loadbalance
}

kube-proxy

kube-proxy runs in host network mode and programs iptables rules to route Service traffic to backend pods. It watches Services, Endpoints, and EndpointSlices and syncs iptables whenever they change.

iptables Chains

ChainPurpose
KUBE-SERVICESEntry point. Matches ClusterIP:port and jumps to per-service chains.
KUBE-SVC-*Per-service chain. Load-balances across endpoints using probability-based rules.
KUBE-SEP-*Per-endpoint chain. DNATs to the actual pod IP:port.
KUBE-NODEPORTSMatches NodePort traffic and jumps to the service chain.
KUBE-MARK-MASQMarks packets for SNAT (masquerade) when leaving the node.
KUBE-POSTROUTINGApplies masquerade to marked packets.
KUBE-FORWARDAccepts forwarded traffic. Includes RELATED,ESTABLISHED rules for return traffic.
State-based sync

kube-proxy uses an order-independent hash (XOR) of all service/endpoint state to detect changes. If the hash matches the previous sync, rules are not flushed and rebuilt. This avoids a brief "connection refused" window that would occur from full flushes every sync cycle. XOR is used because etcd returns items in arbitrary order — an order-dependent hash would cause unnecessary rebuilds.

Traffic Flow: ClusterIP

        

          
          
          
          iptables flow
        

        # Client pod sends to ClusterIP 10.96.0.42:80

PREROUTING → KUBE-SERVICES
  → match 10.96.0.42/80 → KUBE-SVC-XXXX
    → probability 0.33 → KUBE-SEP-AAA (DNAT to 10.244.0.5:8080)
    → probability 0.50 → KUBE-SEP-BBB (DNAT to 10.244.1.3:8080)
    → probability 1.00 → KUBE-SEP-CCC (DNAT to 10.244.1.7:8080)

POSTROUTING → KUBE-POSTROUTING
  → masquerade if marked

Session Affinity

When sessionAffinity: ClientIP is set, kube-proxy adds --sticky iptables rules using the recent module to route repeat connections from the same source IP to the same backend for the configured timeout (default 10800 seconds).

Filter Table Rules

kube-proxy also programs the filter table's FORWARD chain to accept traffic routed through the cluster. The KUBE-FORWARD chain must include rules to accept RELATED,ESTABLISHED return traffic and a jump from the OUTPUT chain for locally-originated traffic.

NetworkPolicy

NetworkPolicies control traffic flow between pods. By default, all pods can communicate with all other pods. Once a NetworkPolicy selects a pod, only explicitly allowed traffic is permitted.

Key Spec Fields

FieldTypeDescription
podSelectorLabelSelectorPods this policy applies to. Empty = all pods in namespace.
policyTypesarray["Ingress"], ["Egress"], or ["Ingress", "Egress"].
ingressarrayAllowed inbound rules. Each rule has from (sources) and ports.
egressarrayAllowed outbound rules. Each rule has to (destinations) and ports.

Peer Selectors

Each from or to entry can use one or more of these selectors. Within a single entry, selectors are ANDed. Across entries, they are ORed.

SelectorDescription
podSelectorMatch pods by labels in the same namespace.
namespaceSelectorMatch pods in namespaces with matching labels.
ipBlockMatch IP ranges. Fields: cidr (required), except (excluded ranges).

Example

        

          
          
          
          networkpolicy.yaml
        

        apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: web-policy
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: web
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              role: frontend
        - namespaceSelector:
            matchLabels:
              env: production
      ports:
        - protocol: TCP
          port: 80
    - from:
        - ipBlock:
            cidr: 10.0.0.0/8
            except:
              - 10.0.1.0/24
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: db
      ports:
        - protocol: TCP
          port: 5432
    - to:                       # Allow DNS
        - namespaceSelector: {}
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
      
        

          
          
          
          deny-all.yaml
        

        # Default deny all ingress and egress in a namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
  namespace: default
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress

Ingress & IngressClass

An Ingress defines HTTP(S) routing rules that map hostnames and paths to backend Services. An IngressClass specifies which controller handles the Ingress.

IngressClass

        

          
          
          
          ingressclass.yaml
        

        apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx

Ingress

        

          
          
          
          ingress.yaml
        

        apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-ingress
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - app.example.com
      secretName: app-tls
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web
                port:
                  number: 80
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api
                port:
                  number: 8080
  defaultBackend:
    service:
      name: default-backend
      port:
        number: 80

Path Types

pathTypeDescription
ExactCase-sensitive exact match of the URL path.
PrefixMatch by URL path prefix, split by /.
ImplementationSpecificMatching depends on the IngressClass controller.
 
        

          
          
          
          terminal
        

        # Create an Ingress
kubectl create ingress web --rule="app.example.com/=web:80" --class=nginx

# List Ingresses
kubectl get ingress

# Describe for status and events
kubectl describe ingress web-ingress

ServiceCIDR & IPAddress

ServiceCIDR resources define the IP ranges available for ClusterIP allocation. The default range is 10.96.0.0/12. IPAddress resources track individual IP allocations.

  name: default

  cidrs:

CNI (Container Network Interface)

Rusternetes implements the standard CNI specification (v0.4.0+). The kubelet executes CNI plugin binaries from /opt/cni/bin/ using configuration from /etc/cni/net.d/ to set up pod networking.

Default Configuration

The cluster ships with a bridge CNI config providing basic pod networking:

  • Pod CIDR: 10.244.0.0/16
  • Service CIDR: 10.96.0.0/12
  • Cluster DNS: 10.96.0.10 (CoreDNS)
  • Plugins: bridge + host-local IPAM + portmap + firewall

Third-Party CNI Plugins

Any standard CNI plugin works on Linux with network namespace support. Drop the plugin binaries in /opt/cni/bin/ and the config in /etc/cni/net.d/.

PluginTypeFeatures
CalicoBGP routingFull NetworkPolicy support, IP-in-IP / VXLAN overlay, BGP peering
CiliumeBPFL7 network policy, service mesh, DNS-based policies, observability
FlannelOverlaySimple VXLAN overlay, minimal configuration, widely supported
Weave NetMesh overlayAutomatic mesh networking, encryption, multicast
MultusMeta-pluginMultiple network interfaces per pod (SR-IOV, MACVLAN, etc.)

Installing Calico

        

          
          
          
          Terminal
        

        # Download Calico CNI binaries
curl -L https://github.com/projectcalico/cni-plugin/releases/latest/download/calico-amd64 \
  -o /opt/cni/bin/calico
curl -L https://github.com/projectcalico/cni-plugin/releases/latest/download/calico-ipam-amd64 \
  -o /opt/cni/bin/calico-ipam
chmod +x /opt/cni/bin/calico /opt/cni/bin/calico-ipam

# Create Calico CNI config
cat > /etc/cni/net.d/10-calico.conflist <<EOF
{
  "name": "k8s-pod-network",
  "cniVersion": "0.3.1",
  "plugins": [
    {
      "type": "calico",
      "ipam": { "type": "calico-ipam" },
      "policy": { "type": "k8s" }
    },
    { "type": "portmap", "capabilities": { "portMappings": true } }
  ]
}
EOF

Docker Compose Integration

Mount CNI binaries and config into the kubelet containers: 

        

          
          
          
          docker-compose.yml
        

        kubelet:
  volumes:
    - /opt/cni/bin:/opt/cni/bin:ro
    - ./cni-config:/etc/cni/net.d:ro

macOS Limitation

NOTE

CNI requires Linux network namespaces. On macOS (Docker Desktop / Podman Machine), the kubelet automatically detects this limitation and falls back to Docker’s bridge networking. Pods still get IPs and can communicate, but CNI plugins are not invoked.

Troubleshooting CNI

  • Pods stuck in ContainerCreating — Check kubelet logs for CNI errors: docker logs rusternetes-kubelet 2>&1 | grep -i cni
  • Pods can’t reach each other — Verify pod IPs with kubectl get pods -o wide and test connectivity with kubectl exec pod-a -- ping <pod-b-ip>
  • CNI not initializing — The kubelet tests network namespace support on startup. If it fails, it logs a warning and falls back to Docker networking.

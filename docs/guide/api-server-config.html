Configuration › API Server

API Server Configuration

The API server is the central hub of the Rusternetes cluster. It exposes the Kubernetes REST API over HTTPS using Axum with 76 handlers and a 2500-line router.

CLI Flags

Flag Type Default Description
--bind-address String 0.0.0.0:6443 Address and port the API server listens on
--etcd-servers String http://localhost:2379 Comma-separated list of etcd endpoints
--log-level String info Log level: trace, debug, info, warn, error
--jwt-secret String rusternetes-secret-change-in-production HMAC fallback secret for SA tokens (RS256 preferred when SA keys exist)
--tls bool false Enable TLS/HTTPS
--tls-cert-file String — Path to TLS certificate file (PEM format)
--tls-key-file String — Path to TLS private key file (PEM format)
--tls-self-signed bool false Generate a self-signed certificate if no TLS files are provided
--tls-san String localhost,127.0.0.1 Comma-separated Subject Alternative Names for self-signed cert
--skip-auth bool false Disable authentication and authorization (insecure, dev only). Enabled by default in all-in-one mode.
--storage-backend String etcd Storage backend: etcd or sqlite
--data-dir String ./data/rusternetes.db SQLite database path (only used with --storage-backend=sqlite)
--prometheus-url String — Prometheus server URL for custom metrics queries (HPA, etc.)

Usage Examples

        

          
          
          
          terminal
        

        # Production with etcd and TLS certificates
rusternetes-api-server \
  --tls \
  --tls-cert-file /etc/rusternetes/certs/api-server.crt \
  --tls-key-file /etc/rusternetes/certs/api-server.key \
  --etcd-servers http://etcd:2379

# Development with self-signed cert
rusternetes-api-server \
  --tls \
  --tls-self-signed \
  --skip-auth

# SQLite backend (no external dependencies)
rusternetes-api-server \
  --storage-backend sqlite \
  --data-dir ./data/rusternetes.db \
  --tls --tls-self-signed

HTTP/2 Settings

The API server configures HTTP/2 parameters to match Kubernetes behavior and support many concurrent watch streams without flow control stalls:

Setting Value Purpose
Initial stream window size 256 KB Per-stream flow control window (K8s: 256 KB)
Initial connection window size 25 MB Total connection flow control (256 KB × 100 streams)
Max concurrent streams 250 Maximum HTTP/2 streams per connection
ALPN protocols h2, http/1.1 Required for Go client-go compatibility
HTTP/2 ALPN

The TLS configuration must set ALPN protocols to ["h2", "http/1.1"]. Without this, Go's client-go library cannot negotiate HTTP/2, causing watch streams to fail with "context canceled" errors.

TLS Configuration

Self-Signed Certificate

When --tls-self-signed is used, the API server generates an in-memory certificate with these properties:

Property Value
Validity 10 years
Organization Rūsternetes
Country US
CA TRUE
SANs From --tls-san flag (default: localhost,127.0.0.1)

PEM Certificate Loading

For production, provide PEM-encoded certificate and key files. The API server supports mTLS — the loaded certificate is also used to verify client certificates when presented.

Authentication

The API server uses RS256 JWT tokens for ServiceAccount authentication. It automatically detects RSA keys at .rusternetes/certs/sa.key and sa.pub. If RSA keys are not found, it falls back to HMAC HS256 using the --jwt-secret value.

CHANGE THE DEFAULT SECRET

The default --jwt-secret value is insecure. In production, always use RS256 with proper SA signing keys, or at minimum change the HMAC secret.

Startup Behavior

On startup, the API server:

  • Connects to the storage backend (etcd or SQLite)
  • Initializes the TokenManager with RS256 keys or HMAC fallback
  • Sets up RBAC authorization (or AlwaysAllow if --skip-auth)
  • Bootstraps the kubernetes Service Endpoints with the server's IP
  • Creates the default ServiceCIDR (10.96.0.0/12)
  • Pre-allocates ClusterIPs from existing services to prevent collisions
  • Starts the HTTP/2 server with TLS (or plain HTTP if TLS disabled)

High Availability

Run Rūsternetes in a highly available configuration with a 3-node etcd cluster, leader election for the scheduler and controller manager, and multiple API server instances behind a load balancer. Tolerate the loss of any single component without cluster downtime.

NOTE

High availability requires the etcd storage backend. The SQLite/Rhino backend does not support replication or multi-writer setups. See the Deployment Overview for a comparison of all modes.

3-Node etcd Cluster

A 3-node etcd cluster tolerates the failure of one node while maintaining quorum (2/3). The compose.ha.yml file defines three etcd instances with static bootstrapping.

etcd ports

Port Protocol Purpose
2379 HTTP Client API. The API server connects to all three endpoints: http://etcd-1:2379,http://etcd-2:2379,http://etcd-3:2379. The client automatically fails over if one node is down.
2380 HTTP Peer communication. Used for Raft consensus, log replication, and leader election between etcd nodes. Must be reachable between all etcd members.

Key etcd settings

Setting Value Description
initial-cluster-token rusternetes-etcd-cluster Unique token to distinguish this cluster from other etcd clusters on the same network. All members must use the same token.
initial-cluster-state new Set to new for initial bootstrap. Change to existing when adding a member to an already-running cluster.
auto-compaction-retention 10m Compacts history older than 10 minutes. Critical for Kubernetes workloads that generate many watch revisions.
quota-backend-bytes 8 GB Maximum database size per member. etcd stops accepting writes when any member exceeds this limit.
snapshot-count 5000 Transactions between snapshots. Lower values increase disk I/O but speed up node recovery after crashes.

Leader Election

The scheduler and controller manager use etcd-based leader election to ensure only one instance of each component is actively reconciling at any time. Standby instances monitor the lease and take over if the leader fails.

How it works

  1. Each instance creates or renews a lease in etcd at a well-known key.
  2. The instance that holds the lease is the leader and actively runs reconciliation loops.
  3. Standby instances watch the lease key and attempt to acquire it if the leader's lease expires.
  4. The leader renews its lease at lease-duration / 3 intervals (default: every 5 seconds for a 15-second lease).
  5. If a leader crashes, the lease expires after the full lease duration, and a standby takes over.

Scheduler leader election flags

Flag Type Default Description
--enable-leader-election bool false Enable leader election. Required when running multiple scheduler instances. When disabled, the scheduler assumes it is the sole instance.
--leader-election-identity String "scheduler-1" Unique identity for this scheduler instance. Must be different for each instance in the HA setup. Used to identify the current leader in logs and lease metadata.
--leader-election-lock-key String "/rusternetes/scheduler/leader" etcd key used for the leader election lease. All scheduler instances must use the same key to compete for the same lock.
--leader-election-lease-duration String "15s" Duration of the leader lease. If the leader fails to renew within this period, another instance can take over. Longer values reduce failover speed but tolerate more network jitter.

Controller manager leader election flags

The controller manager uses the same flags as the scheduler with different default values for identity and lock key:

Flag Type Default Description
--enable-leader-election bool false Enable leader election for the controller manager.
--leader-election-identity String "controller-manager-1" Unique identity for this controller manager instance.
--leader-election-lock-key String "/rusternetes/controller-manager/leader" etcd key for the controller manager leader lease.
--leader-election-lease-duration String "15s" Leader lease duration.

Lock key format

Leader election lock keys follow the format: 

        

          
          
          
          lock key format
        

        
/rusternetes/{component}/leader

# Examples:
/rusternetes/scheduler/leader
/rusternetes/controller-manager/leader

Timing

Parameter Value Description
Lease duration 15 seconds Time before a lease expires if not renewed. This is the maximum failover time.
Renew interval 5 seconds (lease/3) How often the leader renews its lease. Calculated as lease-duration divided by 3.
Retry interval 2 seconds How often standby instances attempt to acquire the lease after the leader's lease expires.
Worst-case failover ~17 seconds Lease duration (15s) + retry interval (2s). In practice, failover is faster because standbys watch for lease expiry.
TIP

You can check which instance is the current leader by reading the lease key from etcd: etcdctl get /rusternetes/scheduler/leader. The value contains the leader's identity string and lease expiration timestamp.

Example: HA scheduler configuration

        

          
          
          
          compose.ha.yml (scheduler instances)

    image: rusternetes-scheduler
    command:
      - --etcd-servers=http://etcd-1:2379,http://etcd-2:2379,http://etcd-3:2379
      - --enable-leader-election
      - --leader-election-identity=scheduler-1
      - --leader-election-lock-key=/rusternetes/scheduler/leader
      - --leader-election-lease-duration=15s

      - rusternetes-net

  scheduler-2:
    image: rusternetes-scheduler
    command:
      - --etcd-servers=http://etcd-1:2379,http://etcd-2:2379,http://etcd-3:2379
      - --enable-leader-election
      - --leader-election-identity=scheduler-2
      - --leader-election-lock-key=/rusternetes/scheduler/leader
      - --leader-election-lease-duration=15s

      - rusternetes-net

Multi-Master API Server

Unlike the scheduler and controller manager, the API server is stateless and can run multiple instances simultaneously without leader election. All instances connect to the same etcd cluster and serve identical responses.

Load balancer setup

Place a load balancer (HAProxy, Nginx, or a cloud LB) in front of multiple API server instances. The load balancer should:

  • Terminate TLS or pass through HTTPS to the API servers
  • Health-check each API server at /healthz
  • Route traffic using round-robin or least-connections
  • Support HTTP/2 (required for watch streams and long-lived connections)
        

          
          
          
          haproxy.cfg
        

        
frontend kubernetes-api
    bind *:6443 ssl crt /etc/haproxy/certs/server.pem alpn h2,http/1.1
    default_backend api-servers

backend api-servers
    balance roundrobin
    option httpchk GET /healthz
    http-check expect status 200

    server api-1 api-server-1:6443 check ssl verify none alpn h2
    server api-2 api-server-2:6443 check ssl verify none alpn h2
    server api-3 api-server-3:6443 check ssl verify none alpn h2
WARNING

The load balancer must support HTTP/2 for watch API streams to work correctly. Without HTTP/2, long-lived watch connections may time out or fail to deliver events. Ensure ALPN negotiation includes h2.

Failure Scenarios and Recovery

Single etcd node failure

Aspect Impact
Quorum Maintained (2/3 nodes). Cluster continues to accept reads and writes.
API server No impact. The etcd client fails over to a healthy endpoint automatically.
Recovery Restart the failed etcd node. It rejoins the cluster and replays the Raft log to catch up.
Data loss None. Data is replicated across all 3 nodes before acknowledging writes.

Two etcd node failure (quorum lost)

Aspect Impact
Quorum Lost (1/3 nodes). Cluster becomes read-only — no writes are accepted.
API server Read requests succeed. Write requests (create, update, delete) fail.
Recovery Restart at least one failed node to restore quorum (2/3). If data is corrupted, restore from snapshot.
Data loss None if at least one node's data is intact. If all data is lost, restore from etcdctl snapshot backup.

Scheduler or controller manager leader failure

Aspect Impact
Failover time Up to 17 seconds (lease duration + retry interval).
During failover New pods remain in Pending (scheduler) or controllers pause reconciliation (controller manager). No data loss.
Recovery Automatic. The standby instance acquires the lease and resumes operations. No manual intervention needed.

API server instance failure

Aspect Impact
Failover time Immediate. The load balancer detects the health check failure and stops routing to the failed instance.
Client impact Active watch streams on the failed instance are disconnected. Clients reconnect automatically to another instance.
Recovery Restart the failed API server. The load balancer adds it back when the health check passes.

Kubelet failure

Aspect Impact
Node status Node transitions to NotReady after the node controller's grace period (default 40 seconds).
Pod impact Pods on the failed node are not evicted immediately. After the eviction timeout (default 5 minutes), pods are rescheduled to healthy nodes.
Recovery Restart the kubelet. It re-registers with the API server and resumes managing its pods.

Monitoring HA Health

        

          
          
          
          terminal
        

        
# Check etcd cluster health
podman compose -f compose.ha.yml exec etcd-1 \
  etcdctl --endpoints=http://etcd-1:2379,http://etcd-2:2379,http://etcd-3:2379 \
  endpoint health --write-out=table

# Check etcd cluster member list
podman compose -f compose.ha.yml exec etcd-1 \
  etcdctl member list --write-out=table

# Check current scheduler leader
podman compose -f compose.ha.yml exec etcd-1 \
  etcdctl get /rusternetes/scheduler/leader --print-value-only

# Check current controller manager leader
podman compose -f compose.ha.yml exec etcd-1 \
  etcdctl get /rusternetes/controller-manager/leader --print-value-only

# Watch leader election changes in real time
podman compose -f compose.ha.yml exec etcd-1 \
  etcdctl watch /rusternetes/ --prefix

Best Practices

  • Separate etcd disks — Use dedicated SSDs for etcd data volumes. etcd is latency-sensitive and shared disks can cause leader election instability.
  • Backup etcd regularly — Schedule etcdctl snapshot save at least every hour. Store backups off-cluster.
  • Monitor etcd metrics — Watch for high apply latency (>100ms), large database size (>4 GB), and failed proposals.
  • Use odd numbers of etcd nodes — 3 or 5 nodes. Even numbers (2, 4) provide no additional fault tolerance over one fewer node.
  • Spread across failure domains — Place etcd nodes, API servers, and kubelets on different physical hosts or availability zones.
  • Set appropriate lease durations — Shorter leases (10s) give faster failover but increase etcd write load. Longer leases (30s) reduce overhead but slow failover.
  • Test failover regularly — Periodically stop a component and verify that failover works as expected.
DANGER

Never reduce a 3-node etcd cluster to 1 node in production. If the single remaining node fails, all data is lost unless you have a snapshot backup. Always maintain at least 2 healthy nodes.

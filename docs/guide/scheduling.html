Scheduling & Autoscaling

The Rusternetes scheduler assigns pods to nodes using a filter-then-score plugin pipeline. This page covers node affinity, pod affinity/anti-affinity, taints and tolerations, priority and preemption, topology spread constraints, resource requests and limits, HPA, VPA, and PodDisruptionBudgets.

Node Affinity

Node affinity constrains which nodes a pod can be scheduled on based on node labels. It replaces the older nodeSelector with more expressive rules.

Required vs. Preferred

TypeBehavior
requiredDuringSchedulingIgnoredDuringExecutionHard constraint. Pod will not schedule if no node matches.
preferredDuringSchedulingIgnoredDuringExecutionSoft constraint. Scheduler prefers matching nodes but will schedule elsewhere if needed. Uses weight (1–100) for priority.

Match Expressions

OperatorDescription
InLabel value must be in the specified set.
NotInLabel value must not be in the specified set.
ExistsLabel key must exist (value ignored).
DoesNotExistLabel key must not exist.
GtLabel value must be greater than specified (parsed as integer).
LtLabel value must be less than specified (parsed as integer).
 
        

          
          
          
          node-affinity.yaml
        

        spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: kubernetes.io/os
                operator: In
                values: ["linux"]
              - key: topology.kubernetes.io/zone
                operator: In
                values: ["us-east-1a", "us-east-1b"]
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 80
          preference:
            matchExpressions:
              - key: node-type
                operator: In
                values: ["high-memory"]
        - weight: 20
          preference:
            matchExpressions:
              - key: disktype
                operator: In
                values: ["ssd"]

Pod Affinity & Anti-Affinity

Pod affinity schedules pods near (or far from) other pods based on labels. This is essential for co-locating related services or spreading replicas across failure domains.

Key Differences from Node Affinity

FieldDescription
topologyKeyRequired. Node label key that defines the topology domain (e.g., kubernetes.io/hostname, topology.kubernetes.io/zone).
namespaceSelectorOptional. Scope pod selection to specific namespaces by label.
namespacesOptional. Explicit list of namespaces to search.
 
        

          
          
          
          pod-affinity.yaml
        

        spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: ["cache"]
          topologyKey: kubernetes.io/hostname
          # Schedule on the same node as pods with app=cache
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: web
            topologyKey: topology.kubernetes.io/zone
            # Prefer spreading web pods across zones

Taints & Tolerations

Taints are applied to nodes. Tolerations are applied to pods. A pod can only schedule on a tainted node if it has a matching toleration.

Taint Effects

EffectDescription
NoSchedulePods without matching toleration will not be scheduled. Existing pods are unaffected.
PreferNoScheduleScheduler avoids the node but will schedule there if no other option exists.
NoExecuteExisting pods without matching toleration are evicted. New pods are not scheduled.

Toleration Operators

OperatorBehavior
EqualKey, value, and effect must match. Default operator.
ExistsKey and effect must match. Value is ignored.
 
        

          
          
          
          terminal
        

        # Taint a node
kubectl taint nodes node-1 dedicated=gpu:NoSchedule

# Remove a taint
kubectl taint nodes node-1 dedicated=gpu:NoSchedule-

# Overwrite a taint
kubectl taint nodes node-1 dedicated=gpu:NoExecute --overwrite
      
        

          
          
          
          tolerations.yaml
        

        spec:
  tolerations:
    - key: dedicated
      operator: Equal
      value: gpu
      effect: NoSchedule
    - key: node.kubernetes.io/not-ready
      operator: Exists
      effect: NoExecute
      tolerationSeconds: 300       # Evict after 5 min if node is not ready
    - operator: Exists             # Tolerate everything (DaemonSet pattern)

PriorityClass & Preemption

PriorityClasses assign scheduling priority to pods. Higher-priority pods are scheduled first and can preempt (evict) lower-priority pods when resources are scarce.

Key Fields

FieldDescription
valueInteger priority value. Higher = more important. System classes use 1,000,000,000+.
globalDefaultIf true, this is the default priority for pods without priorityClassName.
preemptionPolicyPreemptLowerPriority (default) or Never (never preempt other pods).
descriptionHuman-readable description.
 
        

          
          
          
          priorityclass.yaml
        

        apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "Critical workloads that can preempt lower-priority pods."
      
        

          
          
          
          pod-with-priority.yaml
        

        spec:
  priorityClassName: high-priority
  containers:
    - name: app
      image: myapp:latest

Preemption Behavior

  • The scheduler identifies the lowest-priority pods on candidate nodes.
  • It evicts enough pods to free resources for the high-priority pod.
  • PodDisruptionBudgets are respected during preemption when possible.
  • Pods with preemptionPolicy: Never wait in the queue instead of evicting others.
        

          
          
          
          terminal
        

        # Create PriorityClass
kubectl create priorityclass high-priority --value=1000000 --description="Critical workloads"

# List PriorityClasses
kubectl get priorityclasses

TopologySpreadConstraints

Topology spread constraints distribute pods evenly across failure domains (zones, nodes, regions) to improve availability.

Key Fields

FieldDescription
maxSkewMax allowed difference in pod counts between topology domains. Default 1.
topologyKeyNode label key defining domains (e.g., topology.kubernetes.io/zone).
whenUnsatisfiableDoNotSchedule (hard, default) or ScheduleAnyway (soft, prefer even spread).
labelSelectorSelect pods to count for spread calculation.
minDomainsMinimum number of eligible domains. Prevents uneven spread when domains < desired.
matchLabelKeysPod label keys to AND with labelSelector (for revision-aware spreading).
 
        

          
          
          
          topology-spread.yaml
        

        spec:
  topologySpreadConstraints:
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule
      labelSelector:
        matchLabels:
          app: web
    - maxSkew: 1
      topologyKey: kubernetes.io/hostname
      whenUnsatisfiable: ScheduleAnyway
      labelSelector:
        matchLabels:
          app: web

Resource Requests & Limits

Containers specify resource requests (used by the scheduler to find a node) and limits (enforced by the kubelet via cgroup limits on the Docker container).

Supported Resources

ResourceUnitsExample
cpuMillicores (m) or cores100m = 0.1 cores, "2" = 2 cores
memoryBytes with suffixes (Ki, Mi, Gi, Ti)128Mi, 1Gi
ephemeral-storageBytes with suffixes1Gi
Extended resourcesInteger quantitiesnvidia.com/gpu: 1, example.com/fakecpu: 2
Extended resources

The scheduler checks extended resources (like nvidia.com/gpu or custom fakecpu) during resource fit filtering. Nodes must advertise these resources in status.capacity and status.allocatable. 

        

          
          
          
          resources.yaml
        

        containers:
  - name: app
    image: myapp:latest
    resources:
      requests:
        cpu: 250m
        memory: 256Mi
        ephemeral-storage: 1Gi
      limits:
        cpu: "1"
        memory: 512Mi
        ephemeral-storage: 2Gi
        nvidia.com/gpu: 1

HorizontalPodAutoscaler (HPA)

HPA automatically scales the number of pods in a Deployment, ReplicaSet, or StatefulSet based on observed metrics.

Key Spec Fields

FieldDescription
scaleTargetRefReference to the resource to scale (kind, name, apiVersion).
minReplicasMinimum replica count. Default 1.
maxReplicasMaximum replica count. Required.
metricsArray of metric targets. Types: Resource, Pods, Object, External.
behaviorScale-up and scale-down policies with stabilization windows and rate limits.
 
        

          
          
          
          hpa.yaml
        

        apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: AverageValue
          averageValue: 500Mi
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Percent
          value: 100
          periodSeconds: 60
        - type: Pods
          value: 4
          periodSeconds: 60
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
      
        

          
          
          
          terminal
        

        # Create HPA from command line
kubectl autoscale deployment web --min=2 --max=20 --cpu-percent=70

# View HPA status
kubectl get hpa

# Detailed view
kubectl describe hpa web-hpa

VerticalPodAutoscaler (VPA)

VPA adjusts container resource requests and limits based on observed usage. Unlike HPA, it changes per-pod resources rather than replica count.

Update Modes

ModeBehavior
OffOnly generate recommendations; do not apply them.
InitialApply recommendations only at pod creation time.
AutoApply recommendations by evicting and recreating pods with updated resources.
 
        

          
          
          
          vpa.yaml
        

        apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: web-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  updatePolicy:
    updateMode: Auto
  resourcePolicy:
    containerPolicies:
      - containerName: "*"
        minAllowed:
          cpu: 100m
          memory: 128Mi
        maxAllowed:
          cpu: "4"
          memory: 4Gi
        controlledResources: ["cpu", "memory"]

PodDisruptionBudget (PDB)

PDBs limit the number of pods that can be simultaneously unavailable during voluntary disruptions (node drain, cluster upgrades, preemption). They do not protect against involuntary disruptions (hardware failures).

Key Spec Fields

FieldDescription
minAvailableMinimum pods that must be available. Can be integer or percentage.
maxUnavailableMaximum pods that can be unavailable. Can be integer or percentage.
selectorLabel selector for pods this PDB protects.
Choose one

Set either minAvailable or maxUnavailable, not both. If neither is set, the PDB defaults to maxUnavailable: 0, which prevents any voluntary disruption. 

        

          
          
          
          pdb.yaml
        

        apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  maxUnavailable: 1
  selector:
    matchLabels:
      app: web
      
        

          
          
          
          terminal
        

        # Create PDB
kubectl create pdb web-pdb --selector=app=web --max-unavailable=1

# View PDB status
kubectl get pdb

# Drain a node (respects PDBs)
kubectl drain node-1 --ignore-daemonsets --delete-emptydir-data

# Cordon a node (prevent new scheduling, don't evict)
kubectl cordon node-1

# Uncordon (re-enable scheduling)
kubectl uncordon node-1

Scheduler Pipeline

The scheduler processes each unscheduled pod through a two-phase pipeline: 

        

          
          
          
          scheduler pipeline
        

        Phase 1: Filter (eliminate infeasible nodes)
  NodeResourcesFit     — Check CPU, memory, extended resources
  NodeAffinity         — Evaluate node affinity rules
  TaintToleration      — Check taints vs. tolerations
  PodTopologySpread    — Enforce maxSkew constraints
  InterPodAffinity     — Evaluate pod affinity/anti-affinity
  VolumeBinding        — Check PVC availability
  NodeUnschedulable    — Skip cordoned nodes
  HostPort             — Reject pods with conflicting hostPorts

Phase 2: Score (rank remaining nodes)
  NodeResourcesFit     — Prefer nodes with balanced resource utilization
  NodeAffinity         — Score preferred affinity matches
  TaintToleration      — Penalize PreferNoSchedule taints
  PodTopologySpread    — Prefer even distribution
  InterPodAffinity     — Score preferred pod affinity matches
  ImageLocality        — Prefer nodes that already have the image

Phase 3: Bind
  Set pod.spec.nodeName to the highest-scoring node

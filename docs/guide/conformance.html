What's Passing

The following areas have full or near-full conformance coverage:

  • Workloads — Deployments, ReplicaSets, StatefulSets, DaemonSets, Jobs, CronJobs
  • Services — ClusterIP, NodePort, headless services, EndpointSlices
  • RBAC — Roles, ClusterRoles, Bindings, ServiceAccount token creation
  • Custom Resources — CRD lifecycle, schema validation, status/scale subresources
  • Storage — PersistentVolumes, PersistentVolumeClaims, StorageClasses, volume mounts
  • Scheduling — Node affinity, pod affinity/anti-affinity, taints/tolerations, priority/preemption
  • Admission — Validating webhooks, mutating webhooks, ValidatingAdmissionPolicy (CEL)
  • ConfigMaps & Secrets — CRUD, volume projection, environment variable injection
  • Namespaces — Lifecycle, finalizers, cascading deletion
  • Events & Logging — Pod log streaming, event recording

Remaining Failures

The 43 remaining failures fall into these categories:

  • CRD OpenAPI schema edge cases — Complex nested schema validation, x-kubernetes-* extensions
  • Webhook edge cases — Timeout handling, failure policy interactions
  • Networking proxy — Pod proxy subresource, port-forwarding edge cases
  • Container restart timing — Docker container start/stop race conditions, restart backoff
All failures have fixes coded

Fixes for all 43 remaining failures have been implemented and are awaiting deployment in the next conformance round. See the full failure tracker for details.

Running Conformance Tests

Prerequisites

  • A running Rūsternetes cluster (Docker Compose or all-in-one)
  • The cluster must be bootstrapped: bash scripts/bootstrap-cluster.sh
  • KUBECONFIG set to ~/.kube/rusternetes-config

Start the conformance suite

        

          
          
          
          terminal
        

        # Run the full conformance lifecycle
bash scripts/run-conformance.sh

This script uses Sonobuoy to run the official Kubernetes e2e conformance test suite against your cluster. The tests take approximately 2–4 hours to complete depending on hardware.

Monitor progress

        

          
          
          
          terminal
        

        # Watch pass/fail counts in real time
bash scripts/conformance-progress.sh

# View raw e2e output (inside the e2e container)
# Log location: /tmp/sonobuoy/results/e2e.log
podman compose exec e2e tail -f /tmp/sonobuoy/results/e2e.log
Use conformance-progress.sh for accurate counts

The conformance-progress.sh script parses the e2e output and provides accurate real-time pass/fail counts. Do not rely on Sonobuoy's built-in status reporting, which can lag behind the actual test execution.

E2e log location

The detailed test output is written to /tmp/sonobuoy/results/e2e.log inside the e2e container. This log contains the full Ginkgo test output, including individual test names, pass/fail status, and failure details with stack traces.

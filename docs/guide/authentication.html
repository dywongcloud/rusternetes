Authentication & Authorization

How to secure your rusternetes cluster. By default, rusternetes runs with --skip-auth which bypasses all authentication and authorization. This guide shows you how to enable proper security with JWT tokens, RBAC, and client certificate (mTLS) authentication.

WARNING: --skip-auth

By default, the all-in-one binary runs with --skip-auth=true. Every request is treated as coming from admin in the system:masters group. No tokens are required. All RBAC checks are bypassed. Do not expose the API server to any network with --skip-auth enabled.

Authentication Flow

When --skip-auth is disabled, the API server authenticates requests through these steps:

        

          
          
          
          Authentication Chain
        

        Request with Authorization: Bearer <token>
    |
    +-- Try ServiceAccount JWT (TokenManager)
    |     Valid? -> UserInfo from JWT claims
    |
    +-- Try Bootstrap Token (BootstrapTokenManager)
    |     Valid? -> UserInfo from bootstrap token
    |
    +-- No token or invalid?
          -> Anonymous (system:anonymous)
    |
    v
RBAC Authorization (Roles + Bindings)

Quick Start: Securing a Cluster

1. Generate signing keys

        

          
          
          
          Terminal
        

        # RSA key pair for JWT token signing
mkdir -p .rusternetes/certs
openssl genrsa -out .rusternetes/certs/sa.key 2048
openssl rsa -in .rusternetes/certs/sa.key \
  -pubout -out .rusternetes/certs/sa.pub

2. Create an admin ServiceAccount

While still running with --skip-auth, create an admin user:

        

          
          
          
          Terminal
        

        # Create admin ServiceAccount
kubectl create serviceaccount cluster-admin -n kube-system

# Bind to cluster-admin ClusterRole
kubectl create clusterrolebinding cluster-admin-binding \
  --clusterrole=cluster-admin \
  --serviceaccount=kube-system:cluster-admin

# Create a long-lived token Secret
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Secret
metadata:
  name: cluster-admin-token
  namespace: kube-system
  annotations:
    kubernetes.io/service-account.name: cluster-admin
type: kubernetes.io/service-account-token
EOF

# Retrieve the token
TOKEN=$(kubectl get secret cluster-admin-token -n kube-system \
  -o jsonpath='{.data.token}' | base64 -d)
echo "$TOKEN"

3. Configure kubectl with the token

        

          
          
          
          Terminal
        

        kubectl config set-credentials rusternetes-admin --token="$TOKEN"
kubectl config set-context rusternetes --user=rusternetes-admin

4. Restart without --skip-auth

        

          
          
          
          Terminal
        

        rusternetes --skip-auth=false --data-dir ./cluster.db

# Verify authentication works
kubectl get pods                      # should work with token
curl -k https://localhost:6443/api/v1 # should get 401/403

Client Certificate Authentication (mTLS)

For Kubernetes-style client certificate auth, use --client-ca-file. The TLS layer will require clients to present a certificate signed by the specified CA.

        

          
          
          
          Terminal
        

        # Generate CA
openssl genrsa -out ca.key 2048
openssl req -new -x509 -key ca.key -out ca.crt \
  -days 3650 -subj "/CN=rusternetes-ca"

# Generate admin client cert (CN=username, O=group)
openssl genrsa -out admin.key 2048
openssl req -new -key admin.key -out admin.csr \
  -subj "/CN=admin/O=system:masters"
openssl x509 -req -in admin.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -out admin.crt -days 365

# Start with mTLS
rusternetes \
  --tls --tls-cert-file server.crt --tls-key-file server.key \
  --client-ca-file ca.crt --skip-auth=false

# Configure kubectl with client cert
kubectl config set-credentials admin \
  --client-certificate=admin.crt \
  --client-key=admin.key

Token Types

Type Format Use Case
ServiceAccount JWT RS256 or HS256 signed JWT Primary auth for kubectl, pods, controllers
Bootstrap Token <id>.<secret> (6+16 chars) Node join / cluster bootstrapping
Anonymous No token No permissions by default; can be granted via RBAC

Security Checklist

Before exposing a rusternetes cluster to any network:

  • Remove --skip-auth from the API server command
  • Generate RSA signing keys for ServiceAccount tokens
  • Enable TLS (--tls with cert/key or --tls-self-signed)
  • Create an admin ServiceAccount and ClusterRoleBinding
  • Verify: kubectl get pods --token=<token> works
  • Verify: curl -k https://localhost:6443/api/v1/pods returns 401/403
  • Create limited-scope ServiceAccounts for non-admin users
  • Consider --client-ca-file for mTLS network access control

CLI Flags

Flag Default Description
--skip-auth true (all-in-one) / false (standalone) Skip authentication and authorization
--jwt-secret default string HMAC secret for JWT (fallback when no RSA keys)
--client-ca-file disabled Client CA cert for mTLS authentication
--tls false Enable TLS/HTTPS
--tls-cert-file — Server certificate (PEM)
--tls-key-file — Server private key (PEM)

Storage Backends

Rusternetes supports three storage backends: etcd for production clusters, SQLite (via rhino) for lightweight deployments, and an in-memory backend for testing. All backends implement the same Storage trait.

Backend Comparison

Feature etcd SQLite (rhino) Memory
Use case Production clusters Lightweight / all-in-one Unit tests only
Persistence Distributed, replicated Single file on disk None (lost on restart)
Watch support gRPC watch streams Broadcast channel (capacity 1000) Broadcast channel
External dependency etcd cluster required None (embedded) or rhino gRPC server None
HA support Yes (etcd replication) No No
Auto-compaction Yes (etcd built-in) Yes (every 300s) N/A
Auto-UID generation No No Yes
Feature flag Always available sqlite Always available

etcd Backend

The production-grade backend. Connects to an external etcd cluster via gRPC and provides full watch support through etcd's native watch API. 

        

          
          
          
          terminal
        

        # Start with etcd (default)
rusternetes-api-server \
  --storage-backend etcd \
  --etcd-servers http://etcd:2379

etcd Configuration

etcd itself is configured separately. Recommended settings for Rusternetes:

Setting Recommended Purpose
--auto-compaction-retention 1 (hour) Automatically compact old revisions to save disk space
--quota-backend-bytes 8589934592 (8 GB) Maximum database size
--snapshot-count 10000 Number of committed transactions to trigger a snapshot
--listen-client-urls http://0.0.0.0:2379 Client-facing listen address 
        

          
          
          
          docker-compose.yml
        

        etcd:
  image: quay.io/coreos/etcd:v3.5.17
  command:
    - etcd
    - --name=etcd0
    - --data-dir=/etcd-data
    - --listen-client-urls=http://0.0.0.0:2379
    - --advertise-client-urls=http://etcd:2379
    - --auto-compaction-retention=1
    - --snapshot-count=10000

SQLite Backend (rhino)

The lightweight backend uses rhino, an embedded SQLite engine that implements the same key-value semantics as etcd. It can run in two modes:

Embedded Mode (All-in-One)

The SQLite database runs in-process with no external dependencies. This is the default for the all-in-one binary. 

        

          
          
          
          terminal
        

        # All-in-one with embedded SQLite
rusternetes \
  --storage-backend sqlite \
  --data-dir ./data/rusternetes.db
Setting Value Description
compact_interval 300s How often to compact old revisions
Watch mechanism Broadcast channel In-process tokio broadcast for watch events
Feature flag sqlite Must be enabled at compile time

gRPC Mode (Docker Compose)

For multi-container deployments without etcd, rhino can run as a separate gRPC service that multiple components connect to. Use docker-compose.sqlite.yml: 

        

          
          
          
          terminal
        

        # Start with SQLite gRPC backend
docker compose -f docker-compose.sqlite.yml up -d

Memory Backend

An in-memory HashMap-based storage backend used exclusively for unit tests. It provides all Storage trait operations including watch support via broadcast channels. Data is lost when the process exits. 

        

          
          
          
          rust test
        

        // Unit tests use MemoryStorage
#[tokio::test]
async fn test_create_pod() {
    let storage = Arc::new(MemoryStorage::new());
    // MemoryStorage auto-generates UIDs for resources
    let pod = storage.create("/registry/pods/default/test", &pod).await?;
}
TESTING ONLY

The memory backend is not suitable for any production or development use. It has no persistence, no compaction, and auto-generates UIDs which differs from the production backends.

Storage Trait

All backends implement the Storage trait, which provides the complete API for resource persistence:

Method Description
create(key, value) Create a new resource. Returns the created resource with resourceVersion set.
get(key) Get a resource by key. Returns 404 if not found.
update(key, value) Update an existing resource. Returns the updated resource.
update_raw(key, value) Update with raw JSON value (used by garbage collector).
delete(key) Delete a resource by key.
list(prefix) List all resources matching a key prefix.
watch(prefix) Watch for changes to resources with a given prefix.
watch_from_revision(prefix, revision) Watch for changes starting from a specific revision.
current_revision() Get the current storage revision number.
is_revision_compacted(revision) Check if a revision has been compacted and is no longer available.

Key Schema

All backends use the same key format, matching Kubernetes conventions: 

        

          
          
          
          key format
        

        # Namespaced resources
/registry/{resource_type}/{namespace}/{name}

# Cluster-scoped resources
/registry/{resource_type}/{name}

# Examples
/registry/pods/default/nginx          # Pod "nginx" in namespace "default"
/registry/services/kube-system/kube-dns  # Service "kube-dns" in kube-system
/registry/nodes/node-1                # Node "node-1" (cluster-scoped)
/registry/clusterroles/admin          # ClusterRole "admin"

# List prefixes
/registry/pods/default/               # All pods in "default" namespace
/registry/pods/                       # All pods in all namespaces
/registry/nodes/                      # All nodes

Resource Versions

The metadata.resourceVersion field is derived from the storage backend's modification revision:

  • etcd: resourceVersion = etcd mod_revision (monotonically increasing integer)
  • SQLite: resourceVersion = rhino mod_revision (same semantics)
  • Memory: resourceVersion = internal counter

Resource versions are used for optimistic concurrency control (conflict detection on updates) and as watch resume tokens (watch_from_revision).

CHOOSING A BACKEND

Use etcd for production multi-node clusters that need HA and replication. Use SQLite for single-node or development deployments where simplicity matters. Use memory only in test code.

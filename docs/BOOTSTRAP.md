# Cluster Bootstrap Process

This document explains how Rusternetes clusters are bootstrapped with ServiceAccounts, tokens, and CoreDNS.

## Overview

The bootstrap process sets up essential cluster resources:
1. **ServiceAccounts**: Default service accounts for `default` and `kube-system` namespaces
2. **Tokens**: Unique authentication tokens for each ServiceAccount (generated dynamically)
3. **CoreDNS**: Cluster DNS service for pod name resolution
4. **Namespaces**: `default` and `kube-system` namespaces
5. **Services**: Kubernetes API service endpoint

## Bootstrap Script

The main bootstrap script is `scripts/bootstrap-cluster.sh`. It orchestrates the complete bootstrap process:

```bash
./scripts/bootstrap-cluster.sh
```

### What it does:

1. **Generates ServiceAccount tokens** by calling `scripts/generate-default-serviceaccounts.sh`
   - Creates unique, random tokens using `openssl rand -base64 64`
   - Outputs to `.rusternetes/default-serviceaccounts.yaml` (gitignored)

2. **Applies ServiceAccounts and Secrets** to the cluster
   - Creates `default` ServiceAccount in `default` namespace
   - Creates `default` ServiceAccount in `kube-system` namespace
   - Creates token Secrets associated with each ServiceAccount

3. **Applies bootstrap resources** from `bootstrap-cluster.yaml`
   - Creates `default` and `kube-system` namespaces
   - Creates `kubernetes` service for API server discovery
   - Creates CoreDNS pod and service for cluster DNS

4. **Waits for CoreDNS** to become ready
   - Polls CoreDNS pod status for up to 60 seconds
   - Provides diagnostics if CoreDNS fails to start

## Integration with Dev Scripts

The bootstrap process is automatically integrated into development workflows:

### macOS Development (dev-setup-macos.sh)

When you run **"Full setup (build + start)"** (option 8):
```bash
./scripts/dev-setup-macos.sh
# Choose option 8
```

The script will:
1. Build container images
2. Start the cluster
3. **Automatically run `bootstrap-cluster.sh`**
4. Display cluster status

### Fedora Development (dev-setup-fedora.sh)

The generated helper script `.dev/bootstrap.sh` calls the main bootstrap script:
```bash
cd /path/to/rusternetes
./.dev/bootstrap.sh
```

Or use the convenience function:
```bash
source .env.dev
cluster-bootstrap
```

### Conformance Testing (run-conformance.sh)

The conformance test runner automatically bootstraps CoreDNS if it's not running:
```bash
./scripts/run-conformance.sh
```

If CoreDNS is not found or not running, the script:
1. Deletes the old CoreDNS pod (if exists)
2. Calls `bootstrap-cluster.sh` to recreate it with fresh tokens

## Security Considerations

### Generated Tokens

- Tokens are generated using cryptographically secure random data (`openssl rand`)
- Each bootstrap run creates **new unique tokens**
- Tokens are stored in `.rusternetes/default-serviceaccounts.yaml`
- This file is **gitignored** and should never be committed

### Production vs Development

⚠️ **Important**: The tokens generated by `generate-default-serviceaccounts.sh` are:
- NOT properly signed JWTs
- For development and testing purposes only
- Should NOT be used in production

In production Kubernetes:
- ServiceAccounts are created automatically by the ServiceAccount controller
- Tokens are properly signed JWTs created by the TokenController
- Token rotation and expiration are managed automatically

## Manual Bootstrap

If you need to bootstrap the cluster manually:

```bash
# 1. Ensure the cluster is running
docker-compose up -d
# or
podman-compose up -d

# 2. Run the bootstrap script
./scripts/bootstrap-cluster.sh

# 3. Verify cluster resources
export KUBECONFIG=.rusternetes/kubeconfig
./target/release/kubectl get pods -A
./target/release/kubectl get serviceaccounts -A
```

## Troubleshooting

### CoreDNS not starting

If CoreDNS fails to start after bootstrap:

1. Check kubelet logs:
   ```bash
   docker-compose logs kubelet
   ```

2. Check CoreDNS pod status:
   ```bash
   ./target/release/kubectl get pod coredns -n kube-system -o yaml
   ```

3. Check if ServiceAccount token exists:
   ```bash
   ./target/release/kubectl get secret default-token -n kube-system
   ```

4. Re-run bootstrap:
   ```bash
   ./scripts/bootstrap-cluster.sh
   ```

### ServiceAccount tokens not found

If pods can't find ServiceAccount tokens:

1. Verify the tokens were generated:
   ```bash
   ls -la .rusternetes/default-serviceaccounts.yaml
   ```

2. Re-generate tokens:
   ```bash
   ./scripts/generate-default-serviceaccounts.sh
   ```

3. Apply the tokens:
   ```bash
   ./target/release/kubectl apply -f .rusternetes/default-serviceaccounts.yaml
   ```

### Bootstrap script not found

If `bootstrap-cluster.sh` is not found:

1. Ensure you're in the project root directory
2. Make the script executable:
   ```bash
   chmod +x scripts/bootstrap-cluster.sh
   ```

## Files

- `scripts/bootstrap-cluster.sh` - Main bootstrap orchestration script
- `scripts/generate-default-serviceaccounts.sh` - Token generation script
- `bootstrap-cluster.yaml` - Static bootstrap resources (namespaces, CoreDNS, services)
- `.rusternetes/default-serviceaccounts.yaml` - Generated ServiceAccounts and tokens (gitignored)
